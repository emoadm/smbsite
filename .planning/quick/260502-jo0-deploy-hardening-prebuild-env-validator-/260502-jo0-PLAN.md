Create `scripts/check-env.ts` with the following behavior (no new dependencies — use Node built-ins + `tsx` which is already a devDependency):

1. Read all `.ts` and `.tsx` files under `src/` recursively (use `node:fs/promises` `readdir({ withFileTypes: true, recursive: true })` available in Node 20+).
2. Extract every distinct identifier matching the regex `/process\.env\.(NEXT_PUBLIC_[A-Z0-9_]+)/g` from those files. Strip out matches inside line/block comments (use a simple state machine OR strip `^\s*\*` / `^\s*//` / `/\*...\*/` regions before matching — see note below).
3. Read `.env.example` from repo root, parse to a `Set<string>` of keys (lines matching `/^([A-Z_][A-Z0-9_]*)=/`).
4. Compute three sets:
   - `usedInSrc`: NEXT_PUBLIC_* names found in src/ scan
   - `inExample`: NEXT_PUBLIC_* names present as keys in .env.example
   - `inEnv`: NEXT_PUBLIC_* names present AND non-empty in `process.env`
5. Fail (process.exit(1)) with a clearly-formatted multi-line error if EITHER:
   - **Drift**: `usedInSrc \ inExample` is non-empty (src/ references a NEXT_PUBLIC_* var that is not declared in .env.example) — print "ADD TO .env.example: <names>".
   - **Missing/empty in env**: `usedInSrc \ inEnv` is non-empty (src/ uses a NEXT_PUBLIC_* var that is missing or empty in current process.env) — print "MISSING IN ENV: <names>" + the file:line(s) where each is referenced (re-emit the grep hits collected in step 2).
6. PASS silently (`console.log('OK: NEXT_PUBLIC_* env validation passed (N vars checked)')` + exit 0) when both sets are empty. Vars present in .env.example but NOT in src/ (e.g. `NEXT_PUBLIC_PLAUSIBLE_DOMAIN`) MUST NOT cause failure regardless of whether they are empty in env — that is the explicit allowance for "declared-but-not-yet-wired" vars.

- Top-level script, no main() wrapper needed.
- Use `await` at top level (tsx supports it).
- Comment-stripping: the safe minimal approach is `content.replace(/\/\*[\s\S]*?\*\//g, '').replace(/^\s*\/\/.*$/gm, '')` BEFORE applying the NEXT_PUBLIC_ regex. Keep the original (non-stripped) `content` plus its line numbers when emitting the failure report so the error message points to real source lines.
- Skip the `node_modules`, `.next`, `dist` directories defensively, even though scanning is rooted at `src/`.
- When reading .env.example, ignore lines starting with `#` and blank lines.

<task type="auto">
  <name>Task 2: GET-based post-deploy smoke gate in deploy.yml</name>
  <files>.github/workflows/deploy.yml</files>
  <action>
Append a new step to the END of the existing `deploy` job in `.github/workflows/deploy.yml` (do NOT create a new job — the smoke must run inside the same job, AFTER `flyctl deploy` exits 0, so there is no circular dependency on its own success).

The new step must use plain `bash` + `curl` (no extra GitHub Actions, no extra deps). Behavior:

1. Define the probe targets: `/`, `/register`, `/login` against `https://chastnik.eu`.
2. For each target, retry up to 3 times with `sleep 10` between attempts to allow the rolling deploy to settle (machine restart + readiness window). On each attempt:
   - Run `curl -sS -o /tmp/body -w "%{http_code} %{size_download}\n" -L https://chastnik.eu${path}`
     - `-L` follows redirects so `/` (which 307s to `/register`) lands on a populated body, not just a redirect header.
     - `-sS` silent but show errors.
     - `-w "%{http_code} %{size_download}"` captures final status + final body bytes.
   - Parse `${http_code}` and `${size_download}`.
   - PASS condition for the attempt: `http_code` matches `^(2|3)[0-9][0-9]$` AND `size_download -gt 100`.
   - If pass → break the retry loop and move to next path.
   - If fail and attempts remain → sleep 10 and retry.
   - If all 3 attempts fail → echo a clear failure message naming the path, last status, last byte count, and `cat /tmp/body | head -c 500` so the failure is debuggable in Action logs, then `exit 1`.
3. After all 3 paths pass, echo `OK: post-deploy smoke gate passed (3 paths × ≤3 attempts).` and exit 0.

YAML name: `Post-deploy smoke (GET / /register /login, body > 100 bytes)`.

The step body should be a single `run: |` block (avoid splitting into multiple steps to keep retry state simple). Use a small bash function or for-loop. Example skeleton (executor — adapt syntax, this is illustrative only):

```bash
set -uo pipefail
BASE="https://chastnik.eu"
fail=0
for path in "/" "/register" "/login"; do
  ok=0
  for attempt in 1 2 3; do
    out=$(curl -sS -o /tmp/body -w "%{http_code} %{size_download}" -L "${BASE}${path}" || echo "000 0")
    code=${out%% *}
    size=${out##* }
    if [[ "$code" =~ ^(2|3)[0-9][0-9]$ ]] && [ "$size" -gt 100 ]; then
      echo "OK ${path} (attempt ${attempt}): ${code} ${size}B"
      ok=1; break
    fi
    echo "FAIL ${path} (attempt ${attempt}/3): code=${code} size=${size}B"
    [ "$attempt" -lt 3 ] && sleep 10
  done
  if [ "$ok" -ne 1 ]; then
    echo "::error::Smoke failed for ${path} after 3 attempts. Last body (first 500B):"
    head -c 500 /tmp/body || true; echo
    fail=1
  fi
done
[ "$fail" -eq 0 ] || exit 1
echo "OK: post-deploy smoke gate passed."
```

Rationale (DO NOT remove the comment block from the YAML step — copy the key sentences in as a `# ...` header so future operators understand WHY this exists):
- HEAD (`curl -I`) cannot detect the chastnik-eu-empty-page bug (200 + 0 bytes). Must be GET.
- Body-length > 100 is the cheapest assertion that distinguishes "real HTML" from "empty Response" (the broken-state was 0 bytes; the working-state was 27,000–58,000 bytes — a 100-byte floor is safely below working and safely above empty).
- The retry/grace window prevents flake against rolling deploys: the step runs immediately after `flyctl deploy` returns, but Fly machine readiness can lag a few seconds.
- This step is INSIDE the deploy job (not a separate `needs: deploy` job) so there is no possibility of it gating its own deploy. It only runs when `flyctl deploy` returns exit 0.

Test locally before committing:
  a) Run the bash block manually in a terminal pointed at the live https://chastnik.eu (currently working) — should pass all three paths first try.
  b) Run it pointed at a deliberately-bad URL (e.g. https://chastnik.eu/definitely-does-not-exist) — should fail with the diagnostic block.
  c) Lint the YAML with `yamllint .github/workflows/deploy.yml` if available, or at minimum `cat` the file and visually confirm indentation matches the surrounding steps.

Commit with: `chore(ci): add GET-based post-deploy smoke gate to deploy job`

Push happens in Task 3's commit cluster (push all three commits together at the end).
