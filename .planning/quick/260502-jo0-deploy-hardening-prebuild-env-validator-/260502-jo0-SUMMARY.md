## Decisions Made

See `key-decisions` in frontmatter. Key points:
- Inline `&&` form for prebuild gate over npm `prebuild` lifecycle hook (unambiguous under any invocation path).
- Validator scope is "what src/ uses", not "what .env.example declares" — so `NEXT_PUBLIC_PLAUSIBLE_DOMAIN` (declared, unwired) does not block builds.
- Smoke uses GET with body floor, lives inside the deploy job. HEAD-based gate explicitly rejected by source plan because it cannot detect 200+0byte responses.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking] Refactored `scripts/check-env.ts` from top-level await to async-main wrapper**

- **Found during:** Task 1 (validator first run)
- **Issue:** Plan instructed to use top-level await (citing tsx supports it). In practice, the project has no `"type": "module"` in `package.json`, so tsx defaults to CJS output, and esbuild's CJS target rejects top-level await with three transform errors:
  ```
  /Users/emoadm/projects/SMBsite/scripts/check-env.ts:70:14: ERROR: Top-level await is currently not supported with the "cjs" output format
  ```
- **Fix:** Wrapped the body in `async function main(): Promise<void> { ... }` and added `main().catch((err) => { console.error('check-env.ts crashed:', err); process.exit(2); });` at the end. Preserves all behavior; runs identically under tsx in CJS mode.
- **Files modified:** `scripts/check-env.ts`
- **Verification:** All three plan-specified test cases (PASS, FAIL-missing, allowance for empty PLAUSIBLE_DOMAIN) pass after the wrap; drift detection confirmed by separately removing `NEXT_PUBLIC_TURNSTILE_SITE_KEY` from `.env.example`.
- **Committed in:** `bafeae2` (Task 1 commit; the fix was applied before the first commit so it does not require a separate commit).

## Issues Encountered

- One sandbox-only quirk while dry-running the smoke bash locally: the constrained shell stripped `curl` and `sleep` from `PATH`. Re-ran with `/usr/bin/curl` + `/bin/sleep` and got the expected 3-of-3 pass. **Not a code issue** — the GH Actions runner has both binaries on PATH, so the deploy.yml step is unaffected.
- Local `pnpm build` requires `NEXT_PUBLIC_*` vars to be in shell scope (tsx does not auto-load `.env.production.local`). For the dry run I exported them via `export $(grep '^NEXT_PUBLIC_' .env.production.local | xargs -I{} echo {})`. This is intended behaviour — the validator's whole point is to fail loudly when those vars are absent, and CI/Docker injects them through `--build-arg`. Documented for future ops.
