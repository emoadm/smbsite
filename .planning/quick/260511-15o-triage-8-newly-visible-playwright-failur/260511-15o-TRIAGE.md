### 1. agenda.spec.ts:25 — "clicking each TOC link scrolls to the matching <h2>"

**CI error** (extracted from log line 191-202):

```
Test timeout of 30000ms exceeded.
Error: page.waitForLoadState: Test timeout of 30000ms exceeded.

> 27 |     await page.waitForLoadState('networkidle');
```

Same timeout on all 3 retries — never advances past line 27. ANCHORS array never iterated.

**Test source (tests/e2e/agenda.spec.ts:25-43):**

```ts
test('clicking each TOC link scrolls to the matching <h2>', async ({ page }) => {
  await page.goto('/agenda');
  await page.waitForLoadState('networkidle');     // ← times out here
  const ANCHORS = [
    'manifest', 'desen-konsensus', 'ikonomika',
    'energetika', 'zemedelie', 'obrazovanie',
    /* ... 12 anchors total */
  ];
  for (const id of ANCHORS) {
    await page.locator(`nav a[href="#${id}"]`).first().click();
    await expect(page.locator(`#${id}`)).toBeInViewport();
  }
});
```

**Implementation:** `src/app/(frontend)/agenda/page.tsx` ships 12 `<h2 id=...>` chapters; sibling `tests/e2e/agenda.spec.ts:16` ("renders h1 + 12 chapter headings") **passes** in 1.3s, so the page itself renders fine. The TOC component (`src/components/landing/TableOfContents.tsx`) renders the expected `nav` + anchor links. The page itself is healthy.

The failure is `networkidle` not settling. The page renders `<CookieBanner/>` (frontend/layout.tsx:33), which inserts:

```tsx
<Script id="cookieyes"
  src={`https://cdn-cookieyes.com/client_data/${siteKey}/script.js`}
  strategy="beforeInteractive" />
```

with CI env `NEXT_PUBLIC_COOKIEYES_SITE_KEY=ci-cookieyes-placeholder-do-not-use`. The script URL with a placeholder key likely either hangs (CDN routes unknown keys to a long retry/backoff) or returns 200 with internal JS that polls — either way, `networkidle` (no requests in flight for 500ms) never fires on `/agenda` in CI.

The same `networkidle` is absent from `agenda.spec.ts:16` (passes) and `agenda.spec.ts:51` (passes) — only `:25` blocks on it.

**Classification:** **config-bug** (primary) / **test-bug** (secondary).

**Why:** The intended page contract is met (TOC + anchors exist), and the `networkidle` wait is overzealous for an SPA whose ground truth is "DOM is hydrated and anchors exist." The CookieYes placeholder URL in CI is the new env quirk that pushed this from "slow" to "30s timeout." Two fixes work, both stable:

**Recommended fix:**
- Replace `await page.waitForLoadState('networkidle');` with `await page.waitForLoadState('domcontentloaded');` (or remove the wait — `page.goto` already awaits `'load'`). This is identical to how passing tests in this file behave.
- Alternative (deeper): set a non-placeholder `NEXT_PUBLIC_COOKIEYES_SITE_KEY=''` in `.github/workflows/ci.yml` so `CookieBanner` returns `null` (its `if (!siteKey) return null` guard on CookieBanner.tsx:27). But the placeholder is intentional ("present-but-bogus") to exercise the production code path, so keep the placeholder and fix the test.

**Confidence:** HIGH — three signals agree (sibling tests on same page pass, `networkidle` is the only difference, CookieBanner is the only network-active script with a placeholder URL in CI).

**Implementation:** `grep -rn 'role="alert"' src/` finds 4 occurrences, none on the public agenda surface:

- `src/components/ui/alert.tsx:30` — shadcn Alert template (only renders where `<Alert>` is invoked)
- `src/components/dsa/ReportContentDialog.tsx:72` — only rendered for logged-in members on the proposals page

The remaining suspect: CookieYes-injected DOM. The `cdn-cookieyes.com/client_data/<key>/script.js` runtime is documented to inject a consent dialog wrapper, and the [WebServer] error log shows `cky-consent-container` CSS appearing in the trace HTML on subsequent agenda runs. **Classification:** **test-bug**.

**Why:** The implementation contract is met (the agenda's draft banner was removed; ROADMAP confirms Phase 02.3 is in scope to remove it, and agenda page.tsx already has it removed). The test's locator is too broad — it catches any future `role="alert"` element (CookieBanner runtime, future toast notifier, future form errors), not the specific draft banner.

**Recommended fix:**
- Scope the assertion to the article region: `await expect(page.locator('article [role="alert"]')).toHaveCount(0);` — matches the actual intent (no draft banner *inside the agenda article*).
- Even better: assert by content, e.g. `await expect(page.getByText('финализиране')).toHaveCount(0);` — the draft banner specifically contained that word per the sibling `:51` test.

**Confidence:** HIGH for diagnosis (no `<Alert>` in agenda source), MEDIUM for root cause of the `1` element (most likely CookieYes runtime DOM under the CI placeholder key, but could be a Sentry/feedback widget — verifiable by opening the trace.zip).

### 5. landing.spec.ts:22 — "landing emits Cache-Control with s-maxage=3600"

**CI error** (log line 594-606):

```
Error: expect(received).toMatch(expected)
Expected pattern: /s-maxage=3600/
Received string:  "private, no-cache, no-store, max-age=0, must-revalidate"

> 26 |     expect(cc).toMatch(/s-maxage=3600/);
```

**Test source (tests/e2e/landing.spec.ts:22-27):**

```ts
test('PUB-02: landing emits Cache-Control with s-maxage=3600', async ({ request }) => {
  const res = await request.get('/');
  expect(res.status()).toBeLessThan(400);
  const cc = (res.headers()['cache-control'] ?? '').toLowerCase();
  expect(cc).toMatch(/s-maxage=3600/);
});
```

**Implementation:**

- `src/app/(frontend)/page.tsx:19` declares `export const revalidate = 3600;`
- `src/app/(frontend)/layout.tsx:32` renders `<Header />`
- `src/components/layout/Header.tsx:12` calls `await auth()` (reads session cookie)

When a Server Component reads cookies/auth, Next.js opts the entire route into dynamic rendering. Per Next.js 15 docs, `revalidate` only emits `Cache-Control: s-maxage=…` for fully-static / ISR-cached responses; dynamic routes get `private, no-cache, no-store, max-age=0, must-revalidate` — exactly the observed value.

The landing page source itself flags this on lines 16-17:

> "anonymous visitors hit Cloudflare cache; authenticated visitors bypass cache (cookie-vary configured in plan 02-07 middleware/runbook)."

So the *intended architecture* is: origin returns dynamic headers, **Cloudflare's edge cache rules** put `s-maxage=3600` (or equivalent edge TTL) on anonymous responses. The test asserts an origin-layer header that intentionally lives at the CDN.

**Classification:** **code-bug** (primary) / **test-bug** (secondary).

**Why:** Two valid corrections:

1. **Test-bug fix:** The PUB-02 contract is "landing fully static / CDN-cached." The CDN layer is where cache lives; testing the origin's `Cache-Control` is testing the wrong layer. Either delete the test or repurpose it to assert response status + a non-personalized body shape (no per-user name in HTML).

2. **Code-bug fix:** Make `/` truly static for anonymous visitors by splitting the Header. Two paths:
   - **Quick:** Add a static `Cache-Control` override in `next.config.ts` `headers()` for `source: '/'` — `'public, s-maxage=3600, stale-while-revalidate=86400'`. This sets the header at the response layer regardless of dynamic detection. Anonymous responses will be cacheable; authenticated responses will set a `Set-Cookie`/`Vary: Cookie` that Cloudflare honours via Plan 02-07 cache rules. **Risk:** authenticated responses also carry the public header, which leaks per-user content if the Vary rule isn't strict. Verify Cloudflare cache rule still keys on session cookie.
   - **Architectural:** Move `auth()` out of `<Header>` for the landing route — render a logged-out static shell and hydrate logged-in chrome client-side. Larger change; defer to a planned phase.

**Recommended fix:** Decision-needed.
- If the priority is **PR #2 green now**, fix the test: assert response 200 + no per-user content in HTML. Keep `s-maxage` testing for an integration test against the deployed Cloudflare zone (separate fixture).
- If the priority is **lock down PUB-02 contract**, ship the `next.config.ts` `headers()` override and verify the existing Plan 02-07 Cloudflare cache rule honours `Cookie`-vary in front.

**Confidence:** HIGH for diagnosis (Next.js 15 behaviour is well-documented; the page comment on lines 16-17 anticipates exactly this), MEDIUM for "which fix is right" — depends on whether Plan 02-07's Cloudflare rules are already shipped or still pending operator config.
