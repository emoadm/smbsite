# Quick 260511-04m: Fix Playwright Standalone webServer — Summary

Repointed the Playwright CI webServer at a new `pnpm start:standalone` script that copies the static assets Next.js's standalone build deliberately omits, then invokes `node .next/standalone/server.js` directly. This closes the regression on PR #2 where `next start` refused to run under `output: 'standalone'` and nothing bound :3000, causing ~37 page-load timeouts.

## Status

**Incomplete.** Task 1 (the code change) is committed and passes all automated verifications. Task 2 — the `checkpoint:human-verify` gate that asks "push and confirm the CI Playwright job actually boots :3000 with this fix" — is still **open**. The fix has not been validated against a real GitHub Actions run yet.

## What Changed

### `package.json`

Added one script, immediately after the existing `"start": "next start",` entry:

```json
"start:standalone": "cp -r .next/static .next/standalone/.next/static && cp -r public .next/standalone/public && node .next/standalone/server.js",
```

- The two `cp -r` calls populate the static-asset directories Next.js's standalone build omits (only server-traced runtime deps are bundled into `.next/standalone/`).
- `&&` short-circuits — if either copy fails, `node server.js` never runs and the GitHub Actions log surfaces the real error instead of masking it as a 120s timeout.
- Original `"start": "next start"` preserved as the local-dev convenience alias.

### `playwright.config.ts` (lines 38-45 of new file)

- `webServer.command` CI branch: `'pnpm start'` → `'pnpm start:standalone'`.
- Comment block above `webServer` rewritten to explain that `pnpm build` produces `.next/standalone/` and we boot the standalone entry directly because `next start` refuses to run under `output: 'standalone'`.
- `url`, `reuseExistingServer`, `timeout`, and the local-dev `pnpm dev` branch are unchanged.
- All `projects[]`, `use`, `reporter`, etc. untouched.

## Why

`next.config.ts` line 9 has `output: 'standalone'` (intentional, required for the Fly.io Docker image — see `CLAUDE.md` "Application hosting"). Next.js refuses to serve via `next start` under that mode and prints:

> `"next start" does not work with "output: standalone" configuration. Use "node .next/standalone/server.js" instead.`

On PR #2, this meant nothing was listening on :3000, Playwright waited the full 120s, and ~37 specs (landing, branding, unsubscribe, plus the rest of the landing/marketing surface) failed as page-load timeouts. The fix invokes the standalone server directly, and copies in the static assets the standalone build deliberately leaves out (otherwise CSS/JS/fonts/og-image/etc. would 404 and pages would render blank).
