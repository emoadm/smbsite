<objective>
Fix the Phase 5 email worker regression that causes Brevo to reject every
Phase 4 notification email with `400 {"code":"missing_parameter","message":"textContent is missing"}`.

Root cause (confirmed by reading `src/lib/email/worker.tsx`): the three Phase 4
handlers — `submission-status-approved` (line 333-342), `submission-status-rejected`
(line 377-386), and `user-suspended` (line 398-407) — all pass `textContent: ''`
(empty string) to `sendBrevoEmail()`. Brevo treats an empty string as a missing
parameter and returns 400. The Phase 1 and Phase 5 handlers already use the
correct pattern: render the React element once with `render(<X />)` for HTML
and once with `render(<X />, { plainText: true })` for the text body (e.g.
worker.tsx lines 61-62, 188-192, 263-267).

Purpose: Unblock Phase 04.1 Wave 6 Smoke 3 (proposal approval/rejection
notifications), which is gated on these emails landing in the submitter's
inbox. Without the fix, the BullMQ job retries exponentially and the user
never receives the status notification — a P1 functional regression.

Output:
  - 3 patched call sites in `src/lib/email/worker.tsx` that mirror the OTP/
    newsletter pattern (element-once, render-twice).
  - One Vitest unit file (`tests/unit/email-worker-text-content.test.ts`)
    that locks the regression by:
      (a) asserting each of the 3 templates renders a non-empty plain-text
          string, and
      (b) grepping worker.tsx to fail CI if any of the 3 handler blocks
          regresses to `textContent: ''` or `textContent: ""`.

Scope guardrails:
  - Do NOT modify `src/lib/email/brevo.ts`. The empty-string is a worker
    call-site bug, not a client bug — `brevo.ts` correctly forwards what
    it receives (operator decision: the fix belongs where the React element
    is in scope so we get a real plain-text rendering, not a synthetic
    fallback that would silently hide future regressions of this shape).
  - Do NOT change `EmailJobPayload`, queue shape, or job retry policy.
  - Do NOT alter the React templates themselves; they already render
    plain-text correctly via `@react-email/render`'s `plainText: true`
    option (verified by Phase 1 and Phase 5 paths, which use the same
    function and same library version).
  - The operator (not Claude) will re-run Phase 04.1 Smoke 3 after deploy.
    This plan stops at "commit + push"; the merge/deploy/smoke loop is the
    parent Phase 04.1 flow's responsibility.
</objective>
