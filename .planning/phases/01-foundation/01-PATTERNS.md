### Server Action Signature (canonical)
```typescript
'use server';
export async function actionName(
  prevState: ActionState,
  formData: FormData,
): Promise<{ ok: boolean; error?: string; fieldErrors?: Record<string, string[]> }> {
  // 1. Extract + validate with Zod (zod-i18n-map for Bulgarian errors)
  // 2. Anti-abuse checks (rate-limit, Turnstile, disposable-email)
  // 3. Business logic
  // 4. Return { ok: true } or { ok: false, error: 'message.key' }
}
```

### Email Queue Pattern (canonical)
- All outbound email goes through `addEmailJob()` in `src/lib/email/queue.ts`
- Never call Brevo SDK directly from a Server Action
- Worker in `src/lib/email/worker.ts` processes jobs asynchronously
- This pattern applies to ALL future phases (Phase 5 newsletter reuses the same worker)

### Append-Only Table Convention (canonical)
- `consents` table: INSERT only, never UPDATE or DELETE at application level
- Future audit tables (`deletion_log`, `moderation_log` in Phases 5–6) follow the same convention

## Shared Patterns (apply to all implementation tasks)

### Pattern 1: Bulgarian strings via next-intl only
**Source:** CONTEXT.md Established Patterns + D-27
**Apply to:** Every component, server action, email template
- Use `const t = useTranslations('namespace')` in client components
- Use `const t = await getTranslations('namespace')` in server components and actions
- Zero hardcoded Cyrillic strings anywhere in `src/` — CI grep check enforces this (RESEARCH.md line 1213)

### Pattern 2: Zod + zod-i18n-map validation
**Source:** RESEARCH.md § Validation Architecture + CLAUDE.md
**Apply to:** All server actions, all form schemas
- Validate all inputs with Zod before any business logic
- Bulgarian error messages via `zod-i18n-map` (returns message keys that `t()` resolves)
- Schema shape: `{ fieldName: z.string().min(1).refine(...) }`

### Pattern 3: Drizzle-only DB access
**Source:** CONTEXT.md Established Patterns
**Apply to:** All server actions and server components that touch the database
- No raw SQL outside `src/db/migrations/`
- Always import `db` from `@/db`

### Pattern 4: No PII in logs or errors
**Source:** RESEARCH.md § GDPR Posture + D-21
**Apply to:** All server actions, worker, middleware
- Pass only `userId` (UUID) to `logger.*` calls
- Sentry `beforeSend` strips `email` and `ip_address`
- Rate-limiter Redis keys use hashed IP, not raw IP

### Pattern 5: Fail closed on all security checks
**Source:** RESEARCH.md § Turnstile (line 591)
**Apply to:** Turnstile verification, rate-limit checks, disposable-email checks
- Any security check failure → reject the request with Bulgarian error message
- Never fail open (do not silently pass on external service degradation)
- Exception: honeypot field → return fake success (do not leak bot signal)
