<objective>
Close UAT Gap **G1** — make `pnpm worker` boot cleanly on a developer's local box without manual env sourcing.

The standalone BullMQ worker process (`scripts/start-worker.ts`) currently has no env-file loader. When invoked via `pnpm worker` (which runs `tsx scripts/start-worker.ts` per package.json), `process.env.DATABASE_URL` is empty unless the operator manually sources `.env.local` first. The first downstream import (`src/lib/email/worker.tsx` → `src/db/index.ts`) evaluates `url.includes('localhost')` at module top-level against `undefined`, throwing:

```
TypeError: Cannot read properties of undefined (reading 'includes')
    at src/db/index.ts:14:29
```

Production (Fly.io) is unaffected because the `worker` process group inherits Fly secrets via the kernel `environ` (no `.env.local` file present, no dotenv side-effect needed). This is dev-time-only and was first surfaced in Phase 5 because newsletter test-send is the first dev workflow that meaningfully exercises the worker locally.

**Two-part fix:**

1. **scripts/start-worker.ts** — add `import 'dotenv/config'` as the very first line, before any project imports. The dotenv package is already a devDependency (16.4.7) and is already used by `scripts/seed-loadtest-otp.ts` with the identical pattern — this is the project's established convention for tsx-run scripts.
2. **src/db/index.ts** — replace the silent non-null assertion (`process.env.DATABASE_URL!`) with an explicit guard that throws a clear, actionable error message. Defence-in-depth: if a future deployment somehow ships without DATABASE_URL set (env-validator gap, missing Fly secret), the operator gets `Error: DATABASE_URL is not set` instead of a confusing TypeError pointing into `.includes()`.

**Note on dotenv-first ordering:** ES modules evaluate imports in source-text order; side-effect imports (`import 'dotenv/config'`) populate `process.env` synchronously during their own module-load phase. Any project import that follows on subsequent lines will then see the loaded vars at its own top-level evaluation. Putting `import 'dotenv/config'` on line 1 of `scripts/start-worker.ts` guarantees it runs before `import { startWorker as startEmailWorker } from '../src/lib/email/worker'`, which in turn pulls in `src/db/index.ts` whose `const url = process.env.DATABASE_URL!` runs at module load.

**Production safety:** `dotenv/config` looks for `.env` in cwd and silently does nothing if no file is present. On Fly.io, the worker process group has no `.env` file; the import is a harmless no-op. The Fly-injected secrets (already in `process.env` when the Node process starts) remain authoritative.

Output: 2 modified source files + 1 new unit test (~110 LOC total). Phase 5 dev-time worker boot is unblocked.
</objective>

From dotenv 16.x docs (Context7-equivalent — confirmed in node_modules):
- `import 'dotenv/config'` looks for `.env` in `process.cwd()` and loads it.
- If no file is found, it silently returns without modifying process.env.
- For .env.local resolution: dotenv@16.x does NOT auto-resolve `.env.local`. Next.js's own dev server resolves `.env.local` via `@next/env`, but standalone tsx scripts do not.
- **For this project** the dev convention is `.env.local` (per the .env.example sibling pattern). Use `dotenv.config({ path: '.env.local' })` if `.env` is absent — OR rely on the fact that Next.js's `next dev` already loads `.env.local` into the shell environment for any subprocess it spawns. The simplest cross-platform fix is to load BOTH: try `.env.local` first, fall back to `.env`.

**Recommended implementation (decision):** use `dotenv.config({ path: '.env.local', override: false })` followed by `import 'dotenv/config'` to also pull `.env`. Override=false ensures Fly's already-injected secrets win. See `<action>` for the exact two-line snippet.

## Notes

**On the structure choice:** I considered combining G1 with G4 (both touch `scripts/start-worker.ts`) but split them because G4 is a `checkpoint:human-action` plan (operator must toggle Upstash dashboard for staging+production) while G1 is fully autonomous. Mixing autonomous + checkpoint in one plan violates the project's plan-structure conventions (per the planner spec). Splitting also lets G1 land immediately without waiting on operator availability.

**On the `import 'dotenv/config'` vs explicit `loadEnv({ path: '.env.local' })` choice:** the project's `seed-loadtest-otp.ts` uses the single-line `import 'dotenv/config'` pattern. The single-line variant resolves `.env` only (not `.env.local`). The project's `.env.example` indicates `.env.local` is the dev convention. The chosen two-line `loadEnv({ path: '.env.local' }) + loadEnv({ path: '.env' })` pattern resolves both, with `override: false` making Fly secrets win in production. This matches Next.js's own resolution order (`.env.local` overrides `.env` in dev; production drops both in favour of the injected env).
