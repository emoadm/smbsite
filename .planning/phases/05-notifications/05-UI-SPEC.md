# Phase 05 — UI Design Contract

> Visual and interaction contract for Phase 5 (Notifications). Generated by gsd-ui-researcher; verified by gsd-ui-checker.

This contract is binding. The planner and executor must follow these tokens, components, copy, motion and email-rendering rules. Where coalition deliverables (channel URLs) are pending, the placeholder mechanism is **structurally locked** even if content is not.

Phase 5 is a feature-extension on top of Phase 1 + Phase 2 + Phase 02.1. **No design tokens are re-litigated**; this spec inherits §3 (Typography), §4 (Color), §5.6 (Footer), §8 (Accessibility), §10 (Motion), §12 (Breakpoints) of `02-UI-SPEC.md` verbatim and only **extends** with Phase 5-specific components, copy, layouts, and the email design contract.

## 2. Spacing Scale

8-point scale (multiples of 4 only). Inherited from Phase 2 `02-UI-SPEC.md` § 2 verbatim. Phase 5 adds NO new spacing tokens.

| Token | Value | Tailwind | Usage |
|-------|-------|----------|-------|
| 2xs | 4px | `gap-1`, `p-1` | Icon-to-label gaps inline |
| xs | 8px | `gap-2`, `p-2` | Compact list spacing, badge padding, switch handle gap |
| sm | 12px | `gap-3`, `p-3` | Form field internal padding, switch+label gap |
| md | 16px | `gap-4`, `p-4` | Default block gap, mobile section pad, preferences row vertical gap |
| lg | 24px | `gap-6`, `p-6` | Card padding, paragraph block gap, /community channel-card padding |
| xl | 32px | `gap-8`, `p-8` | Inter-section gap on cards, composer split-pane gutter |
| 2xl | 48px | `gap-12`, `py-12` | Major section breaks (mobile) |
| 3xl | 64px | `gap-16`, `py-16` | Major section breaks (md+) |
| 4xl | 96px | `py-24` | Hero vertical pad on desktop (not used in Phase 5) |

**Section vertical rhythm on `/community` and `/member/preferences`:** `py-12 md:py-16` between page header and main content; `py-12 md:py-20` between major sections (matches Phase 2 landing rhythm).

**Email-specific spacing (inline styles, not Tailwind utilities):**
- Container max-width: **600px** (universal email standard; Outlook constraint).
- Container padding: 24px on all sides (`padding: 24` in inline style).
- Heading top margin: 0; bottom margin: 16.
- Paragraph (body block) bottom margin: 16; line-height: 1.6.
- Section dividers (`<Hr />`): 32px vertical margin, `border-color: #E2E8F0` (matches `--color-border`).
- Footer block: 16px top padding above the `<Hr />`, 24px between footer paragraph lines.

**Exceptions:**
- Touch targets ≥44×44 CSS px (WCAG 2.5.5). Switch components meet this when their parent row uses `py-3` (12px top+bottom + 24px control = 48px row).
- Composer "Send blast" button uses `size="lg"` (h-11 = 44px) AND a 16px top margin separating it visibly from the live preview pane.

### 3.2 Email typographic ramp (inline-style values)

Email cannot reliably use rem; declare in pixels. Cyrillic-safe fallback chain.

| Role | Size (px) | Weight | Line-height | Font stack (CSS `fontFamily`) |
|------|-----------|--------|-------------|-------------------------------|
| Email subject (h1) | 28 | 800 | 1.2 | `Gilroy, "Roboto", system-ui, -apple-system, "Segoe UI", Tahoma, sans-serif` |
| Email h2 (section heading) | 22 | 800 | 1.25 | same |
| Email h3 (sub-section) | 18 | 600 | 1.3 | `"Roboto", system-ui, -apple-system, "Segoe UI", Tahoma, sans-serif` |
| Email body | 16 | 400 | 1.6 | same as h3 |
| Email greeting line | 16 | 400 | 1.6 | same |
| Topic chip | 12 | 600 | 1.0 | same; letter-spacing `0.04em`; uppercase |
| CTA button | 16 | 600 | 1.0 | same; padding `12px 24px`; border-radius `6px` |
| Footer microcopy | 12 | 400 | 1.5 | same |
| Copyright line | 11 | 400 | 1.4 | same |

**Outlook desktop font fallback:** Outlook 2016+ on Windows ignores `@import` webfonts; the fallback chain MUST resolve to Roboto (next/font emits as `system-ui`) → Tahoma → sans-serif. Gilroy renders ONLY in clients that honor webfont CSS (Apple Mail, Gmail web). This degradation is acceptable per D-17 (single template; topic distinction is cosmetic).

### 3.3 Cyrillic + encoding (D-18 design obligation)

The following glyphs MUST render correctly across **Gmail web, Outlook desktop (Windows), Apple Mail, abv.bg, mail.bg** before any newsletter ships:

`А Б В Г Д Е Ж З И Й К Л М Н О П Р С Т У Ф Х Ц Ч Ш Щ Ъ Ь Ю Я ѝ ж щ ъ ю я`

- `<Head>` MUST contain `<meta httpEquiv="Content-Type" content="text/html; charset=utf-8" />` AND `<meta charSet="utf-8" />` (defense in depth — Pitfall 5 from RESEARCH).
- Unit test: vitest source-grep on `NewsletterEmail.tsx` asserts both meta declarations and `lang="bg"` attribute.
- Render-snapshot test: rendered HTML matches `/[Ѐ-ӿ]/` AND `/charset="utf-8"/`.
- Manual verification gate (extends `02.1-VALIDATION.md` pattern): Outlook desktop, abv.bg, mail.bg are **manual-verify** — UI-SPEC obligates this verification before D-Phase5-prep is unblocked.

4. RadioGroup item checked-state ring + dot
5. Active link text (e.g., "Виж предпочитанията" on /unsubscribed)

**Save semantics (locked recommendation):** **Per-toggle save with optimistic update + Sonner toast confirmation**.
- Each Switch toggle fires a Server Action `updateNewsletterTopic({ topic, granted })` that INSERTs a new `consents` row (append-only, D-09 + Phase 1 D-13).
- Optimistic update: switch flips state immediately; Sonner toast shows "Записано" on success or "Не успяхме да запазим — опитайте пак" on failure (revert state).
- Same pattern for the channel radio (`updatePreferredChannel({ channel })` writes nullable text on `users.preferred_channel`).
- Language radio is disabled in v1 — no Server Action wired.

**Why per-toggle (not atomic Save button):** Matches Phase 02.1's snappy Server Action pattern; lower friction; append-only audit table accommodates the higher write volume natively.

**Empty state:** N/A — every authenticated member always has 4 topic rows (default-on at registration via D-09; legacy `kind='newsletter'` reads as blanket grant per backward-compat). No empty surface to design.

**Error state (server save fails):** Optimistic switch reverts; Sonner toast (destructive variant) shows "Не успяхме да запазим — опитайте пак". No inline error UI; no form-level error block.

### 7.8 Tone enforcement rules

- **Never vocative.** Phase 1 D-26 lock. `Уважаеми/Уважаема` banned in all email templates.
- **Greeting in email always nominative + comma:** `Здравейте, {firstName}.` Fallback `Здравейте.` Never `Драги Иване,` or `Г-н Иванов,`.
- **Pronoun mix accepted (carry-forward from Phase 2 § 7.6):**
  - `/member/preferences`, `/community`, `/unsubscribed` use formal **2nd-person plural** "вие/ви" (institutional tone)
  - Composer (Payload admin) uses informal **2nd-person singular** "ти" — editors are insiders, formality reads stiff in admin
  - The composer's send-CTAs use universal verb form ("Изпрати рекламата", "Планирай изпращане")
- **Email body greeting:** singular formal `Здравейте` (collective polite — not vocative). Body copy is editor-authored Lexical content; composer prompt strings remind editors of nominative-only rule but do not transform editor text.
- **No exclamation marks** in welcome surfaces (Phase 2 § 7.6 carry-forward). The `/unsubscribed` page intentionally has zero `!`.
- **No emojis** in any UI string. Includes email subject lines and editor-composed body — Lexical config strips emoji nodes if pasted; tested via vitest source grep.

### 11.1 Page-level caching

| Page | Caching | Rationale |

### 11.2 Image optimization (email)

- All inline images uploaded via Payload media collection → stored on Bunny.net CDN → referenced by absolute Bunny URL.
- Custom Lexical upload converter emits `width`/`height` HTML attrs (RESEARCH Pattern 5; Pitfall 1).
- Default width = 600 (full-bleed inside email container); height = auto computed from Bunny metadata.
- Max single image weight: 200KB (editor advisory, not enforced; coalition workflow guideline).

### 11.3 Live preview render budget

- Debounce: 500ms after last composer field change.
- Server Action `renderPreview` budget: ≤300ms p95 (React Email render of typical newsletter is ~80-150ms).
- Iframe render: instant once `srcdoc` updates.
- If render fails: iframe shows last-good HTML + Sonner toast `Прегледът не успя да се обнови` (muted, non-destructive — user keeps editing).
