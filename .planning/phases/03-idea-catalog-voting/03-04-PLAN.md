Result discriminated union (exported for client + tests):
```
export type CastVoteResult =
  | { ok: true; action: 'cast' | 'change' | 'noop'; voteEventLogId?: string }
  | { ok: false; reason: 'unauthenticated' | 'email_not_verified' | 'invalid_input' | 'rate_limited' | 'turnstile_required' | 'captcha_failed' | 'idea_not_published' | 'internal_error' };
```

Pipeline (in this exact order):

1. `const session = await auth();` — if no session, return `{ ok: false, reason: 'unauthenticated' }`.
2. Read `userId` from session. Read `users.email_verified_at` for that user via `db.select(...).from(users).where(eq(users.id, userId)).limit(1)`. If null/missing, return `{ ok: false, reason: 'email_not_verified' }`.
3. `const parsed = VoteInput.safeParse(input);` — on failure return `{ ok: false, reason: 'invalid_input' }`.
4. `const h = await headers(); const ip = getClientIp(h);`
5. `const gate = await gateVoteRate(userId, !!parsed.data.turnstileToken);`
   - If `gate.ok === false && gate.reason === 'hard-block'` → return `{ ok: false, reason: 'rate_limited' }`.
   - If `gate.ok === false && gate.reason === 'turnstile-required'`:
     - If no token → return `{ ok: false, reason: 'turnstile_required' }`.
     - Else `const tr = await verifyTurnstile(parsed.data.turnstileToken, ip ?? undefined);` — if `!tr.ok` return `{ ok: false, reason: 'captcha_failed' }`.
6. Verify the idea is published (defense — published ideas are the only ones the catalog renders, but the action MUST not allow voting on draft/archived ideas):
   ```
   const ideaRow = await db.execute(sql`SELECT status FROM ideas WHERE id = ${parsed.data.ideaId} LIMIT 1`);
   const status = (ideaRow.rows[0] as { status?: string })?.status;
   if (status !== 'published') return { ok: false, reason: 'idea_not_published' };
   ```
7. `const fresh = await isFreshAccount(db, userId);` — capture for the audit row (cooling does NOT block the cast per D-04 — only blocks display).
8. DB transaction (single):
   - Look up existing vote: `tx.select({ id: votes.id, choice: votes.choice }).from(votes).where(and(eq(votes.user_id, userId), eq(votes.idea_id, parsed.data.ideaId))).limit(1)`
   - If existing.choice === parsed.choice → return `{ ok: true, action: 'noop' }` (no DB write — idempotency)
   - Compute `actionKind = existing.length > 0 ? 'change' : 'cast'`
   - UPSERT votes:
     ```
     await tx.insert(votes).values({ user_id: userId, idea_id: parsed.data.ideaId, choice: parsed.data.choice }).onConflictDoUpdate({ target: [votes.user_id, votes.idea_id], set: { choice: parsed.data.choice, updated_at: new Date() } });
     ```
   - INSERT vote_events_log:
     ```
     const ipHash = ip ? hashIp(ip) : '';
     const subnetHash = ip ? hashSubnet(ip) : '';
     const uaHash = hashUa(h.get('user-agent') ?? '');
     const [logRow] = await tx.insert(vote_events_log).values({
       user_id: userId,
       idea_id: parsed.data.ideaId,
       choice: parsed.data.choice,
       action: actionKind,
       ip_hash: ipHash,
       subnet_hash: subnetHash,
       ua_hash: uaHash,
       fresh_account_at_event: fresh,
     }).returning({ id: vote_events_log.id });
     return { ok: true as const, action: actionKind, voteEventLogId: logRow.id };
     ```

9. After the transaction returns successfully, enqueue an anomaly-check job (fire-and-forget, NOT awaited inside the action — Phase 1 D-19 forbids synchronous Brevo / Sentry from Server Actions; anomaly detection runs in the BullMQ worker per plan 03-08):
   ```
   import { addEmailJob } from '@/lib/email/queue';
   void addEmailJob({ kind: 'vote-anomaly-check', ideaId: parsed.data.ideaId }).catch((err) => {
     logger.warn({ event: 'vote_anomaly_enqueue_failed', err_kind: err.constructor.name }, 'Failed to enqueue anomaly check');
   });
   ```
   Note: plan 03-08 extends `EmailJobKind` with `'vote-anomaly-check'` and `'vote-anomaly-alert'`. Until 03-08 ships, this `addEmailJob` call type-errors — that's expected dependency ordering. Plan 03-08 must complete before this Server Action runs in production.

10. Logging:
    - `logger.info({ event: 'vote_cast', user_id: userId, idea_id: parsed.data.ideaId, action: actionKind, fresh_account_at_event: fresh }, 'vote cast');`
    - NEVER log raw IP, raw UA, choice, email, full_name (Phase 1 D-21 PII discipline). The hash columns + the actionKind are sufficient.

11. ABSOLUTE PROHIBITION: do NOT import or call `revalidatePath` or `revalidateTag` from `next/cache`. The catalog count cache is intentionally lazy (D-03). Test `tests/unit/server-action-no-revalidate.test.ts` (plan 03-01) enforces.
