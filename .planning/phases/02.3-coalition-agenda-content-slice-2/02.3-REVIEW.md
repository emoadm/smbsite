# Phase 02.3: Code Review Report

**Reviewed:** 2026-05-10
**Depth:** standard
**Files Reviewed:** 5
**Status:** issues_found

## Summary

Phase 02.3 ships nine new chapters into `/agenda`, extending the locked TOC from 3 to 12, removes the slice-1 `<Alert>` draft banner, cleans the `agenda` i18n namespace, and adds a Playwright spec. The structural deliverables are all present and the unit tests will pass against the current source. No security vulnerabilities, no data-loss risks, no broken anchor wiring (TOC ↔ `<h2 id>` map is 1:1 across all 12 sections).

That said, the slice ships several real defects that would surface in production: a known UX regression where anchor scroll lands chapter headings underneath the site's sticky header (no `scroll-margin-top` configured on `<h2>`), an a11y mislabeling that places two "Основна навигация" landmarks on the same page, multiple `<h3>` elements containing paragraph-length text rather than headings, mixed Bulgarian/ASCII smart-quote pairs across the prose, and a handful of typos copied in from the source PDF. Test scaffolding has correctness issues of its own — one regression guard does cwd-relative IO, another has assertions that pass even if the regex silently captured an empty block, and the Playwright spec passes whether or not the scroll-offset bug is present (`toBeInViewport` with default 0% threshold).

These are quality and UX defects, not blockers, but several should be fixed before this is held up as the canonical agenda page.

## Warnings

### WR-01: Anchor scroll lands `<h2>` headings behind the sticky site header

**File:** `src/app/(frontend)/agenda/page.tsx:85,154,248,626,816,947,1096,1207,1414,1577,1702,1885`
**Issue:** `src/components/layout/Header.tsx:17` renders the global header as `sticky top-0 z-40`. None of the 12 `<h2 id="…">` chapter headings carry `scroll-mt-*` / `scroll-margin-top`, so when the user clicks a TOC link the heading scrolls *under* the sticky header and the user lands mid-paragraph with no visible chapter title. This is a real UX regression from slice 1, where 3 anchors shipped with the same problem but the impact was minor; with 12 anchors and a featured TOC the defect is unavoidable. Note also that `tests/e2e/agenda.spec.ts:41` uses `toBeInViewport()` with the default 0%/1px threshold — the assertion passes even when the heading is 95% obscured by the header — so this bug is not caught by the new E2E spec.
**Fix:**
```tsx
// Apply on every chapter heading (or via the article-level prose styles):
<h2 id="manifest" className="scroll-mt-24">НИЕ СМЕ СИНЯ БЪЛГАРИЯ!</h2>

// Or globally for prose articles in src/app/globals.css:
.prose :where(h2[id], h3[id]) { scroll-margin-top: 6rem; }
```
Also tighten the E2E assertion so it would catch the regression:
```ts
await expect(page.locator(`#${id}`)).toBeInViewport({ ratio: 0.9 });
```

### WR-02: Two `<nav aria-label="Основна навигация">` landmarks on a single page

**File:** `src/app/(frontend)/agenda/page.tsx:69,78`
**Issue:** Both the mobile and desktop `TableOfContents` invocations are passed `tA11y('primaryNavLabel')` ("Основна навигация" / "Primary navigation"). In `TableOfContents.tsx:68-70` the desktop variant renders `<nav aria-label={label}>`. The site already uses "Основна навигация" semantics for the global site header (the string lives in `messages/bg.json:300` under `a11y.primaryNavLabel`). Result: a screen-reader user lands on `/agenda` and is told there are two "Основна навигация" landmarks — one is the site nav, the other is the in-page TOC. They are not the same thing, and the duplicate labelling makes both harder to target via landmark navigation. The page-TOC needs its own a11y string ("Съдържание" / "На тази страница" / "Table of contents").
**Fix:** Add a dedicated message and use it for the TOC:
```jsonc
// messages/bg.json (a11y namespace)
"tableOfContentsLabel": "Съдържание"
```
```tsx
// agenda/page.tsx
const tA11y = await getTranslations('a11y');
const tocLabel = tA11y('tableOfContentsLabel');
// pass tocLabel instead of tA11y('primaryNavLabel') to both <TableOfContents> calls
```

### WR-03: Multi-sentence paragraphs marked as `<h3>` headings

**File:** `src/app/(frontend)/agenda/page.tsx:958-962, 1014-1018, 1323-1327`
**Issue:** Three `<h3>` elements contain full paragraphs (3+ sentences), not headings. Example at line 958-962: `<h3>Нашата цел в областта на училищното образование е да повишим образователните стандарти и да създадем образователна система, която е източник на национална гордост и увереност. Всички предвидени мерки са координирани и целят постигането на тази мисия.</h3>`. This breaks the document outline (an h3 should be a brief label), inflates the screen-reader heading list, and harms scannability. Same defect at lines 1014-1018 (висше образование intro) and 1323-1327 (болнична помощ intro).
**Fix:** Convert each to an `<h3>` with a short title plus a following `<p>`:
```tsx
<h3>Цел на училищното образование</h3>
<p>
  Нашата цел в областта на училищното образование е да повишим
  образователните стандарти и да създадем образователна система, която е
  източник на национална гордост и увереност. Всички предвидени мерки са
  координирани и целят постигането на тази мисия.
</p>
```
If the prose intentionally has no h3 here, drop the heading altogether and keep only the `<p>`.

### WR-04: Mixed Bulgarian opening quote `„` paired with ASCII closing quote `"`

**File:** `src/app/(frontend)/agenda/page.tsx:674, 1046, 1088, 1217, 1282, 1354, 1356, 1407, 1431, 1439, 1707, 1719, 1734, 1765, 1782, 1783, 1787, 1812, 1832` (19 occurrences)
**Issue:** Bulgarian typographic convention pairs `„` (U+201E) with `"` (U+201C), not `"` (ASCII U+0022). The page mixes the two: e.g. line 674 `"политически некоректни"`, line 1217 `"медицински пустини"`, line 1431 `"Синя България"`. The opening quote is set correctly but the closing quote is the ASCII straight double-quote. This is a typographic quality defect on a flagship public page. It is also inconsistent — the site's other Bulgarian copy in `messages/bg.json` uses ASCII `"` throughout — so the agenda page is half-converted.
**Fix:** Either (a) keep ASCII straight quotes everywhere — replace the 19 opening `„` with `"` for internal consistency — or (b) commit to typographic Bulgarian quotes by replacing each ASCII closing `"` with `"` (U+201C). Option (b) is the publisher-correct choice; pick one and apply consistently across the file.

### WR-05: Bulgarian-text typos shipped in chapter prose

**File:** `src/app/(frontend)/agenda/page.tsx:179, 739, 1416, 1708, 1762`
**Issue:**
- Line 179: `безмислени` → should be `безсмислени` ("meaningless")
- Line 739: `Сина България` → should be `Синя България` (brand name typo)


### WR-06: `agenda-i18n-cleanup.test.ts` performs cwd-relative IO and silently passes if regex fails to match

**File:** `tests/unit/agenda-i18n-cleanup.test.ts:10,13`
**Issue:** Two test-quality defects:
1. `readFileSync('messages/bg.json', 'utf8')` is resolved against `process.cwd()`, not against `__dirname`. If vitest is invoked from a subdirectory (e.g. via an editor task or a workspace runner), the test fails with `ENOENT` rather than running. Compare with `tests/unit/agenda-toc-anchors.test.ts:13` which correctly uses `resolve(__dirname, '../../...')`.
2. The regex `/^  "agenda":\s*\{[\s\S]*?\n  \}/m` is brittle: if the JSON is reformatted with tabs or 4-space indent (or a future agenda namespace nests deeper than the current 3 keys), `bg.match(...)?.[0] ?? ''` returns the empty string, and four of the five assertions (`not.toContain('"body"')`, `not.toContain('"draftAlert"')`, `not.toContain('[ТЕКСТ ОТ КОАЛИЦИЯ]')`, `toContain('"metadataDescription"')` — wait, this last one would fail) become false-positives or noisy. The single guard `expect(agendaBlock).toContain('"agenda"')` would catch the empty-string case, but only because `'""agenda""'` contains the substring — actually no, the empty string does NOT contain `"agenda"`, so the guard would catch a total miss. However a *partial* match (e.g. capturing only `"agenda": {` and stopping at the next `}` because of stray indentation) would silently dump truncated content and produce false negatives on the `body`/`draftAlert` guards.
**Fix:**
```ts
import { resolve } from 'node:path';

const bg = readFileSync(
  resolve(__dirname, '../../messages/bg.json'),
  'utf8',
);
const parsed = JSON.parse(bg);
const agenda = parsed.agenda as Record<string, unknown>;

it('agenda namespace exists', () => {
  expect(agenda).toBeDefined();
  expect(agenda.title).toBeDefined();
});

it('agenda.body has been dropped', () => {
  expect(agenda).not.toHaveProperty('body');
});

it('agenda.draftAlert has been dropped', () => {
  expect(agenda).not.toHaveProperty('draftAlert');
});

it('agenda.metadataDescription is set and non-empty', () => {
  expect(typeof agenda.metadataDescription).toBe('string');
  expect((agenda.metadataDescription as string).length).toBeGreaterThan(0);
});

it('no [ТЕКСТ ОТ КОАЛИЦИЯ] placeholder remains in agenda values', () => {
  for (const v of Object.values(agenda)) {
    if (typeof v === 'string') expect(v).not.toContain('[ТЕКСТ ОТ КОАЛИЦИЯ]');
  }
});
```
This eliminates the regex, the cwd dependency, and the false-positive risk in one pass.

### WR-07: Sequential `<ol>` lists inside `Икономика` restart numbering, producing a confusing 1–9 → 1–7 sequence

**File:** `src/app/(frontend)/agenda/page.tsx:320-400, 434-621`
**Issue:** The `ikonomika` chapter renders two adjacent `<ol>` blocks with no separator heading between them. Each `<ol>` restarts at 1, so the reader sees items 1-9 (state-out-of-commerce policy) followed by items 1-7 (budget/tax/growth). There's no visual or semantic cue that these are two distinct lists; the second list's `<li><strong>Бюджетна политика — Дългове и дефицити.</strong></li>` is item "1" again, which reads as if items 1-9 were re-enumerated. The reader also has no chapter sub-heading (`<h3>`) to anchor to.
**Fix:** Either insert an `<h3>Бюджетна и икономическа политика</h3>` (or similar) between the two lists to delineate them, or merge into a single `<ol>` continuing 1-16. If you want continuous numbering across two distinct lists, use `start={10}` on the second `<ol>`:
```tsx
<ol start={10}>
  <li><strong>Бюджетна политика — Дългове и дефицити.</strong> …</li>
  …
</ol>
```

### IN-02: Playwright test relies on Chromium default viewport ≥ md (768px)

**File:** `tests/e2e/agenda.spec.ts:25-43`
**Issue:** The TOC-link click test selects `nav a[href="#${id}"]`, which only matches the desktop variant of `TableOfContents`. The desktop nav has `hidden md:block` (visible only ≥ 768px). The test passes today because Playwright's Chromium default is 1280×720, but if anyone changes the project's `playwright.config.ts` to a phone viewport (or runs against the `'Mobile Safari'` project named in `responsive.spec.ts`), this spec breaks because the locator resolves to zero elements.
**Fix:** Pin the viewport explicitly so the intent is documented in the test:
```ts
test.use({ viewport: { width: 1280, height: 800 } });
```
Or write a parallel mobile-flow spec that opens the `<details>` first and then clicks.

### IN-03: `TableOfContents` `useEffect` dependency `items` is a fresh array on every render

**File:** `src/components/landing/TableOfContents.tsx:46`
**Issue:** The component is invoked twice from `agenda/page.tsx` with `items={TOC_ITEMS}` (a module-level constant — fine), but the dep array `[items]` is a referential check. For server components passing the same constant on every render this is stable, so no extra observers are created in practice. Worth flagging only because if a caller ever inlines the array literal (e.g. `items={[{...}]}`), each render would tear down and reinstall the IntersectionObserver. Not a defect today; defensive code would dep-track on `items.map(i => i.id).join(',')` or similar.
**Fix:** No code change needed today; consider adding a doc comment on `TableOfContents` warning callers to memoize the `items` prop.

### IN-05: `<section>` elements lack `aria-labelledby` linking them to their headings

**File:** `src/app/(frontend)/agenda/page.tsx:84, 153, 246, 624, 814, 945, 1094, 1205, 1412, 1575, 1700, 1883`
**Issue:** A bare `<section>` is *not* exposed as a region landmark in modern AT unless it has an accessible name. Adding `aria-labelledby={h2_id}` would expose each chapter as a navigable region — which is exactly what a "Програма / Agenda" page wants for screen-reader users.
**Fix:**

```tsx
  <h2 id="manifest">НИЕ СМЕ СИНЯ БЪЛГАРИЯ!</h2>

```

### IN-06: `agenda-toc-anchors.test.ts` regex assertion does not check ordering

**File:** `tests/unit/agenda-toc-anchors.test.ts:32-36, 38-42`
**Issue:** The "every TOC id has a matching h2" check uses `expect(h2Ids).toContain(id)`, which validates set inclusion but not order. UI-SPEC §7.2 locks the TOC order; if someone reorders the `<h2>` blocks but leaves the `TOC_ITEMS` array unchanged, the test passes but the page reads out of order. Same with `expect.arrayContaining(...)` in the slice-1/slice-2 anchor tests — order is not asserted.
**Fix:**
```ts
it('TOC and h2 ordering match', () => {
  expect(h2Ids).toEqual(tocIds);
});
```
Drop the multiple `arrayContaining` assertions in favor of one strict-equal against the locked 12-anchor sequence.
