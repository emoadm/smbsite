## Summary

Phase 02.3 is a **content-only extension** of Phase 02.2's walking-skeleton. Phase 02.2 already shipped and locked the architectural contract (hardcoded JSX in `src/app/(frontend)/agenda/page.tsx`, file-level i18n exemption, anchor-drift unit test, 2-column TOC sidebar grid, `TableOfContents` `variant` prop). The Phase 02.3 UI-SPEC (`02.3-UI-SPEC.md`, approved 6/6 PASS) is exhaustive: it locks 18 inherited contracts (L-01 → L-18), specifies every new content shape (`<h3>` sub-sections, em-dash sub-bullets, `<strong>`-prefixed `<ol>` items, etc.), and pre-locks the final 12-entry `TOC_ITEMS` array including exact Cyrillic→ASCII anchor IDs.

Therefore the planner does NOT need to research framework fundamentals, design patterns, or library choices — those are all decided. The remaining unknowns are entirely **content shape** (how to map source lines `319-1332` into JSX) and **slicing strategy** (one plan? grouped chapters? all-in-one?).

The source content `agenda-raw.txt:319-1332` decomposes cleanly into **9 top-level chapters** (one fewer than the description's "~10" estimate, because the UI-SPEC §7.2 final TOC has 9 new entries, not 10) with diverse internal structures: mostly prose+lists, with one chapter (Образование) that contains explicit numbered policies and one (Култура) that contains an explicitly-named "пакет от 10 основни мерки" 10-item `<ol>` and one (Защита на ценностите) that uses `<h3>` sub-divisions. Total estimated rendered output: ~5500-6500 lines of JSX (page.tsx grows from 627 → ~6000-7000 lines).

**Primary recommendation:** Plan as **3 SPIDR slices grouped by structural similarity**, not one-plan-per-chapter. Slice A = simple-prose chapters (Енергетика, Земеделие, Здравеопазване, Външна сигурност); Slice B = numbered-policy chapters (Образование, Социална политика, Култура — the most structurally complex); Slice C = `<h3>`-subsection chapter + final-slice cleanup (Правосъдие, Защита на ценностите, drop draftAlert + drop `agenda.body` key + adopt `agenda.metadataDescription`). This grouping minimizes context-switching for the executor (each slice has consistent shape rules) and keeps each plan under ~2000 lines of new JSX. One-plan-per-chapter would create 9 thin tasks with redundant context; one-plan-for-all would risk context-window exhaustion and mid-task burnout.

## Common Pitfalls

### Pitfall 1: pdftotext list-marker fidelity

**What goes wrong:** The source PDF uses inconsistent visual markers — em-dashes (`–`), bullet dots, indentation-only — for what are semantically `<li>` items. pdftotext output sometimes preserves the marker as a leading character on the line, sometimes drops it, sometimes turns it into whitespace.

**Why it happens:** PDF doesn't distinguish "bullet character" from "list-item structure"; pdftotext extracts visible glyphs. Between chapters, the coalition's PDF used different list styles.

**How to avoid:**
- Per UI-SPEC §3.3.B exception, **strip the source's `–` character from rendered text** when it's the bullet marker (Култура items 6-10). The text starts AFTER the dash.
- Per UI-SPEC §3.3.B, **NEVER introduce `list-disc list-inside` or em-dash markers in JSX** — `prose` supplies the disc.
- For Земеделие (lines 466-518), the source has irregular indentation around bullets — render as a single flat `<ul>` with one `<li>` per discrete policy, even where the source visually splits with a blank line. UI-SPEC §3.3.B Pattern #3.

**Warning signs:**
- Rendered page shows `– text` instead of `• text` → `–` was not stripped → fix
- Rendered page shows nested `<ul>` inside an `<li>` that has no logical sub-list → over-nesting → flatten
- Numerals appear as both `<li>` markers AND text (`1. 1. Policy name`) → hand-coded prefix → strip

### Pitfall 2: pdftotext paragraph wrapping

**What goes wrong:** Single logical paragraphs span 3-10 source lines because the PDF used fixed-width line wrapping. Naive newline-to-`<br>` mapping creates fragmented JSX paragraphs.

**Why it happens:** pdftotext preserves visual line breaks, not paragraph structure.

**How to avoid:** UI-SPEC §3.3.F locks the rule:
- **Blank line in source = paragraph break** → new `<p>`
- **Consecutive non-blank lines = same paragraph** → join into one `<p>`

This was Phase 02.2 ambiguity #2 — already resolved; same rule applies in 02.3.

**Warning signs:** Rendered page shows mid-sentence breaks; readability test "can I read this paragraph aloud naturally?" fails.

### Pitfall 3: Sub-section heading vs. numbered policy ambiguity

**What goes wrong:** In Образование, the source has BOTH (a) a top-level heading "Образование", (b) two numbered sub-sections labeled "1. Нашата цел в областта на училищното образование…" and "1. В сферата на висшето образование…" (yes, BOTH start with "1.") which read like sub-divisions, AND (c) numbered policies inside each sub-section ("1. Промяна на учебното съдържание", "2. Връщане на училището…", "3. Възстановяване и защита на авторитета на учителя"). The planner could conflate these.

**Why it happens:** pdftotext renders both as "1. ...", "2. ...", "3. ..." in plain text, losing the visual hierarchy from the PDF.

**How to avoid:** UI-SPEC §3.3.A is explicit:
- Sub-divisions `<h3>` apply ONLY in **Защита на ценностите** (5 named subsections like "България е свободна и демократична държава") and **Правосъдие** ("Съдебни разходи и правна помощ", "Срокове" — and the planner should also evaluate "Правни средства за защита, достъпни на европейско и международно равнище" at line 1025).
- In Образование, the two "1. Нашата цел…" / "1. В сферата на висшето…" headers are NOT `<h3>` — they're framing sentences. Render them as `<p>` lead-paragraphs OR as `<h3>` if the planner feels they read as section dividers (this is a discretion call — **recommendation:** render them as **`<h3>` without anchor** because the structural shape (a sentence introducing a numbered list) matches §3.3.A's "explicitly named sub-division" pattern).
- Only AFTER each Образование sub-section header come the numbered policies, which DO render as `<ol>` per §3.3.C with `<strong>{policy-name}.</strong>` prefix per §3.3.D.
