## GeoIP source

### Q1 — Which GeoIP resolution method should Phase 2.1 use?

| Option | Description | Selected |
|--------|-------------|----------|
| MaxMind GeoLite2 (self-hosted) | Download .mmdb at build, query in-process via maxmind-db npm. Zero external HTTP per request. Matches legal balancing-test doc. | ✓ |
| ipapi.co or ipinfo.io (hosted REST) | 1 HTTP call per visit. Adds sub-processor (US-incorporated), latency, throttle. CLAUDE.md mentions these but raw IP would leave EU. | |
| Hybrid: GeoLite2 primary + hosted fallback | Self-hosted with hosted fallback for misses. Marginal completeness gain; over-engineered. | |

**User's choice:** MaxMind GeoLite2 (self-hosted)
**Notes:** Resolves the conflict between `attribution-balancing-test.md` (names GeoLite2) and `CLAUDE.md` stack guide (names ipapi.co) in favor of the legal doc.

### Q2 — How should the GeoLite2 .mmdb file be obtained at deploy time?

| Option | Description | Selected |
|--------|-------------|----------|
| Auto-download in Docker build via MAXMIND_LICENSE_KEY | Add license key to Fly secrets + GitHub Actions; RUN curl in Dockerfile. ~80MB image bump. | ✓ |
| Vendor the .mmdb in the git repo | Simple but violates MaxMind EULA (no redistribution). | |
| Use geoip-lite npm package (bundled DB) | Smaller, MIT. Updated less often; weaker BG oblast precision. | |

**User's choice:** Auto-download in Docker build via MAXMIND_LICENSE_KEY
**Notes:** Mirrors Brevo / Sentry secret pattern from Phase 1. Refresh cadence = on each deploy.

### Q3 — What happens when oblast can't be derived (private IP, IPv6 miss, lookup fails)?

| Option | Description | Selected |
|--------|-------------|----------|
| Store oblast = "unknown", country from DB or "unknown" | Always write the row; surface unknown bucket in dashboard. | ✓ |
| Store oblast = NULL, country = NULL when any lookup fails | Same write-always but NULL sentinel. Cleaner schema; needs COALESCE. | |
| Drop the attribution event when oblast lookup fails | Loses 10–30% of visits (VPN/IPv6). Wrong for campaign-attribution. | |

**User's choice:** Store oblast = "unknown"
**Notes:** Matches D-19 fire-and-forget spirit; dashboard shows unknown alongside named oblasts.

## Attribution storage shape

### Q1 — What is the attribution-event recording grain?

| Option | Description | Selected |
|--------|-------------|----------|
| One row per anonymous session | First touch INSERTs; subsequent visits UPDATE last-touch fields. Bounded volume. | ✓ |
| One row per visit (page-view stream) | Highest fidelity but big volume on QR-peak day; needs rollup. | |
| One row per user (only at registration) | Loses pre-reg context; violates SC1 ("every visitor has source recorded"). | |

**User's choice:** One row per anonymous session

### Q2 — What identifies the anonymous session before registration?

| Option | Description | Selected |
|--------|-------------|----------|
| First-party cookie set by middleware | `attr_sid` UUID, 30d, SameSite=Lax, HttpOnly, Secure. Strictly-necessary. | ✓ |
| Server-derived hash (no cookie) | Hash(IP + UA). IP rotation breaks linkage; defeats ATTR-05. | |
| localStorage UUID (client-side) | Fails on JS-disabled visits; harder to read in middleware. | |

**User's choice:** First-party cookie set by middleware

### Q3 — Multiple touches before registration — what do we surface and link?

| Option | Description | Selected |
|--------|-------------|----------|
| Both first-touch and last-touch | Schema stores `first_*` + `last_*` columns. Industry-standard split-attribution. | ✓ |
| Last-touch only | Single column set, UPDATE-overwrite. Loses first-touch causal story. | |
| First-touch only | Cleanest causal story but misses re-engagement signal. | |

**User's choice:** Both first-touch and last-touch

## "Where did you hear" UX

| Option | Description | Selected |
|--------|-------------|----------|

### Q3 — Where does the self-reported answer live in the schema?

| Option | Description | Selected |
|--------|-------------|----------|
| On the user row (self_reported_source + self_reported_other) | Captured at registration submission, never mutated. Cascade-deletes with user. | ✓ |
| On the attribution_events row | Requires UPDATE on attr_sid match at registration; extra complexity. | |
