# Phase 02.1 Plan 02: MaxMind GeoLite2 GeoIP foundation Summary

**One-liner:** In-process MaxMind GeoLite2 lookup wrapper with safe `unknown` fallback, 28-entry Bulgarian oblast lookup table, and a Dockerfile build-secret pipeline that downloads `GeoLite2-City.mmdb` fresh on every deploy without ever committing the EULA-restricted file or leaking the license key into image layers.

## What Shipped

This plan delivered the four foundational pieces consumed by the Wave 2 BullMQ attribution worker (plan 04) and the Wave 4 source-attribution dashboard (plan 07). Wave 1 sequencing in 02.1: this plan is upstream of all attribution-write paths.

1. **`@maxmind/geoip2-node@6.3.4` installed** ŌĆö added to `dependencies` in `package.json` + `pnpm-lock.yaml`. This is the only new runtime npm dependency introduced by Phase 2.1's GeoIP work.

2. **`src/lib/geoip.ts`** ŌĆö Node-runtime-only module with:
   - **Lazy ReaderModel singleton** (`_reader` module-local, mirrors the IORedis singleton pattern in `src/lib/email/queue.ts` lines 16-27). `getGeoReader()` is idempotent.
   - **`lookupIp(ip)`** wrapper with the D-03 fail-safe contract: invalid input or any internal throw returns `{ oblast: 'unknown', country: 'unknown' }` so the downstream worker can ALWAYS write the attribution row. Never throws.
   - **Pitfall 4 mitigation** ŌĆö full ISO 3166-2 oblast code is constructed as `${countryCode}-${subdivSuffix}` so we store `'BG-22'`, not the raw `'22'` numeric suffix MaxMind returns.
   - **Header comment** explicitly forbids importing this module from Edge runtime (`src/middleware.ts` and any Edge route). `Reader.open()` calls `fs.readFile`, which is unavailable in Edge ŌĆö the build would fail with `Module not found: Can't resolve 'fs'`.
   - **`__resetReaderForTests()`** helper exported solely so the unit suite can swap `GEOIP_DB_PATH` between test cases without process-level singleton leak.

3. **`src/lib/oblast-names.ts`** ŌĆö Static `Record<string, string>` lookup table with all 28 Bulgarian oblast codes (`BG-01` ąæą╗ą░ą│ąŠąĄą▓ą│čĆą░ą┤ ŌĆ” `BG-28` ą»ą╝ą▒ąŠą╗) plus the `'unknown' ŌåÆ 'ąØąĄąĖąĘą▓ąĄčüčéąĄąĮ'` fallback. Includes the asymmetric `BG-22` (ąĪąŠčäąĖčÅ) / `BG-23` (ąĪąŠčäąĖčÅ-ą│čĆą░ą┤) entries per ISO 3166-2:BG. Bonus `oblastDisplayName(code)` helper that returns `'ąØąĄąĖąĘą▓ąĄčüčéąĄąĮ'` for null / undefined / unrecognized codes (defensive default).

4. **`tests/unit/geoip.test.ts`** ŌĆö 7 vitest assertions split across two `describe` blocks:
   - **D-03 fallback** ŌĆö empty IP, null input (TS-suppressed runtime guard test), and a malformed IP with a non-existent `GEOIP_DB_PATH` all resolve to `{ oblast: 'unknown', country: 'unknown' }`. None throw.
   - **D-24 oblast names** ŌĆö table has exactly 29 keys; `BG-22 === 'ąĪąŠčäąĖčÅ'`; `BG-23 === 'ąĪąŠčäąĖčÅ-ą│čĆą░ą┤'`; every `BG-01..BG-28` key resolves to a truthy string; `oblastDisplayName(null|undefined|'XX-99')` all return `'ąØąĄąĖąĘą▓ąĄčüčéąĄąĮ'`.

5. **`Dockerfile` ŌĆö builder stage** got a two-step block right after `RUN pnpm build`:
   - `RUN apk add --no-cache curl` ŌĆö node:20-alpine ships without curl, and we use it just-in-time on the builder (the runner does not need curl at runtime).
   - `RUN --mount=type=secret,id=MAXMIND_LICENSE_KEY` reading the secret from `/run/secrets/MAXMIND_LICENSE_KEY`, downloading the GeoLite2-City tar.gz from `download.maxmind.com`, stripping `--strip-components=1 --wildcards "*/GeoLite2-City.mmdb"`, and `mv`'ing the result to `/app/GeoLite2-City.mmdb`. The secret never enters an image layer (verified by the BuildKit secret-mount contract ŌĆö `docker history --no-trunc <image>` will not surface the key).

6. **`Dockerfile` ŌĆö runner stage** gained a single `COPY --from=builder /app/GeoLite2-City.mmdb ./GeoLite2-City.mmdb` line right after the existing `COPY --from=builder /app/package.json` line. The path matches the default in `src/lib/geoip.ts` (`process.cwd() + '/GeoLite2-City.mmdb'`), so no env var override is required at runtime.

7. **`.env.example`** documents both new env vars: `MAXMIND_LICENSE_KEY=` (build-time secret only ŌĆö NOT needed at runtime) and `GEOIP_DB_PATH=` (runtime override; defaults to `${cwd}/GeoLite2-City.mmdb`). Comments explain the dual-secret-store requirement (Fly secrets + GitHub Actions secret).

8. **`.github/workflows/deploy.yml`** ŌĆö the existing `flyctl deploy --strategy rolling --remote-only` command in the `deploy` job now passes `--build-secret "MAXMIND_LICENSE_KEY=${{ secrets.MAXMIND_LICENSE_KEY }}"` immediately before the existing `--build-arg` lines. CI deploys pick up the latest mmdb on every run (no scheduled refresh job needed for v1 per D-02).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 ŌĆö Type Error] ReaderModel typed import added**

- **Found during:** Task 1 GREEN ŌĆö first `tsc --noEmit` reported `src/lib/geoip.ts(40,29): error TS2339: Property 'city' does not exist on type 'Reader'.`
- **Issue:** The plan's example code (and the RESEARCH.md Pattern 1 snippet) typed the singleton as `Reader | null`. In `@maxmind/geoip2-node@6.3.4`, `Reader` is the static factory class (`Reader.open()`); `Reader.open()` returns a `ReaderModel` instance, which is the type that exposes the `.city(ip)` method.
- **Fix:** Imported `ReaderModel` as a type and changed the singleton + return type from `Reader` to `ReaderModel`. Added a one-line comment noting that `Reader.open() returns a ReaderModel`.
- **Files modified:** `src/lib/geoip.ts`
- **Commit:** Folded into the GREEN commit `f786dba` (the type fix landed before the implementation was committed).

### Curl-Availability Decision (Step 1 of Task 2)

The plan's Task 2 Step 1 instructed `docker run --rm node:20-alpine which curl` to detect curl. Docker is not available in this sandboxed agent environment. Falling back to documented base-image knowledge: `node:20-alpine` does NOT ship curl by default (the existing Dockerfile's runner stage already does `apk add --no-cache postgresql-client tini` for the same reason). Accordingly, I added `RUN apk add --no-cache curl` to the builder stage unconditionally (it precedes the secret-mount RUN). The runner stage was NOT modified ŌĆö the runner does not need curl at runtime, only at build time. Documented in the Dockerfile comment block above the secret-mount RUN.

## Operator Handoff (D-MaxMindLicenseKey)

Before the next deploy that exercises this plan's Dockerfile or worker code path, the operator MUST:

1. **Register a free MaxMind account** at https://www.maxmind.com/en/geolite2/signup
2. **Generate a license key** at https://www.maxmind.com/en/accounts/current/license-key (logged in)
3. **Add the key to Fly secrets** so local `fly deploy` works:
   ```
   fly secrets set MAXMIND_LICENSE_KEY=<your-key> -a smbsite-prod
   ```
4. **Add the key to GitHub Actions repo secrets** at `https://github.com/<org>/<repo>/settings/secrets/actions` as `MAXMIND_LICENSE_KEY` so the CI deploy pipeline can build the image.

If the operator skips either step, `flyctl deploy` will fail at the builder's secret-mount RUN with a clear "secret not provided" error (acceptable surface ŌĆö the deploy fails loudly rather than silently shipping without the mmdb). The runtime code path is gracefully degrading regardless: `lookupIp()` returns `{ oblast: 'unknown', country: 'unknown' }` if the mmdb is missing, so the worker still writes attribution rows and the dashboard simply shows everything as "ąØąĄąĖąĘą▓ąĄčüčéąĄąĮ" until the operator completes the handoff.

For local dev (no Docker, no Fly), the unit tests do not require the mmdb file (they exercise the D-03 fallback path with a non-existent `GEOIP_DB_PATH`). To exercise a real lookup locally, the developer would need to manually download GeoLite2-City.mmdb from MaxMind and point `GEOIP_DB_PATH=/path/to/GeoLite2-City.mmdb` in `.env.local`.
