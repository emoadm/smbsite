## Summary

Phase 02.1 is an instrumentation layer that must ship before the QR mail drop. The core architecture follows a strict Edge → Queue → Node Worker split: Edge middleware captures attribution signals synchronously (header reads only, no I/O), enqueues a job with the raw IP, and sets the `attr_sid` cookie. A Node.js BullMQ worker then performs the CPU-bound `@maxmind/geoip2-node` lookup and writes to Postgres. Raw IP never touches the database; it is ephemeral in Redis.

The three major new capabilities are: (1) the attribution capture pipeline (middleware + worker + `attribution_events` schema), (2) the `self_reported_source` field on the registration form (eight Bulgarian-language options), and (3) the Payload admin custom view for editors. All three extend existing Phase 1 infrastructure — BullMQ queue, Drizzle schema, Payload admin shell — without introducing new services.

The load test (OPS-05) is the hardest gate: k6 must be provisioned on a Hetzner CX21 EU runner, scenario target RPS derived from coalition mail-drop volume, and the full registration + attribution path must hold at 2x peak. Cloudflare cache behavior with the `attr_sid` Set-Cookie response header is a critical integration point: by default Cloudflare bypasses cache for responses containing Set-Cookie; the fix is to set the cookie on a request to a non-cached endpoint OR to move cookie-setting to a separate endpoint so that public page responses do not include Set-Cookie.

**Primary recommendation:** Implement the attribution pipeline as a strict Edge-enqueue / Node-worker split, reusing the existing BullMQ + Upstash pattern verbatim. Move `attr_sid` cookie-setting to the middleware response on a dedicated `/api/attr/init` route (or via a non-cached JS fetch from the landing page) so that public CDN-cached pages remain Set-Cookie-free.

### Pattern 1: @maxmind/geoip2-node Reader Singleton

**What:** Load the mmdb file once at process start into a module-level singleton. Reuse for every lookup.

**When to use:** Every attribution worker job. Must NOT be loaded in Edge middleware.

```typescript
// Source: https://context7.com/maxmind/geoip2-node/llms.txt
// src/lib/geoip.ts (Node runtime only — DO NOT import from Edge middleware)
import { Reader } from '@maxmind/geoip2-node';
import path from 'node:path';

let _reader: Reader | null = null;

export async function getGeoReader(): Promise<Reader> {
  if (_reader) return _reader;
  const mmdbPath = process.env.GEOIP_DB_PATH ?? path.join(process.cwd(), 'GeoLite2-City.mmdb');
  _reader = await Reader.open(mmdbPath, { cache: { max: 1000 } });
  return _reader;
}

export async function lookupIp(ip: string): Promise<{ oblast: string; country: string }> {
  try {
    const reader = await getGeoReader();
    const response = reader.city(ip);
    // ISO 3166-2 subdivision code for Bulgaria: BG-01..BG-28
    const subdivCode = response.subdivisions?.[0]?.isoCode ?? null;
    const countryCode = response.country?.isoCode ?? 'unknown';
    return { oblast: subdivCode ?? 'unknown', country: countryCode };
  } catch {
    return { oblast: 'unknown', country: 'unknown' };
  }
}
```

Key property: `response.subdivisions[0].isoCode` returns the ISO 3166-2 code (e.g. `"22"` for Sofia-grad; combined with country code becomes `"BG-22"`). The `names` object provides human-readable names in multiple locales. [VERIFIED: Context7 / @maxmind/geoip2-node llms.txt]

### Pattern 5: attr_sid Cookie + Cloudflare Cache Interaction

**Critical issue (D-CookieVaryCacheRule):** Cloudflare automatically bypasses its cache when the origin response includes a `Set-Cookie` header (default behavior with "origin cache control" enabled). If middleware sets `attr_sid` in the response to `GET /`, the public landing page will receive `cf-cache-status: BYPASS` on every request, breaking PUB-02 CDN caching. [VERIFIED: https://developers.cloudflare.com/cache/concepts/cache-behavior/]

**Recommended resolution (option A — preferred):** The middleware sets the `attr_sid` cookie on the RESPONSE for all requests. Since Cloudflare's bypass-on-Set-Cookie applies to what the origin returns, the fix is to ensure the public pages that ARE cached do not include Set-Cookie. The approach: only set the `attr_sid` cookie via a dedicated lightweight endpoint (`GET /api/attr/init?sid=...`) that is explicitly NOT cached. The middleware reads the existing `attr_sid` from the request (already set by a prior `/api/attr/init` call or a JS-triggered init fetch). On first visit, the page loads from cache; a tiny client-side fetch to `/api/attr/init` sets the cookie in response; subsequent page requests already carry the cookie.

**Alternative (option B — simpler but slightly less elegant):** The middleware sets `attr_sid` only when the cookie is ABSENT from the request. On first visit, the response to `GET /` will contain `Set-Cookie: attr_sid=...`, causing a cache BYPASS for that first request only. Subsequent requests from the same browser will include the `attr_sid` cookie in the request (not the response), so Cloudflare caches the response normally. In practice this means: first-time QR scan visitors get a cache MISS, return visits are cached. This degrades QR launch performance only for new visitors.

**Planner decision required:** The planner must pick A or B. Option A preserves full CDN caching for all anonymous visitors; option B trades one MISS per new visitor for implementation simplicity.

**Cookie classification in CookieYes:** In the CookieYes dashboard → Cookie Manager → Necessary category → "+ Add Cookie" button. Enter: Cookie ID = `attr_sid`, Domain = `chastnik.eu`, Duration = 30 days, Description = `Бисквитка за анонимна идентификация на сесията за целите на кампанията (необходима).` [VERIFIED: CookieYes docs on cookie classification — manual add to Necessary bucket; users cannot disable Necessary cookies]

### Pattern 6: k6 Load Test (OPS-05)

**Target RPS derivation formula:**
```
mail_drop_volume × expected_scan_rate_pct × peak_compression_factor / peak_window_seconds
```
Example: 10,000 letters × 5% scan rate = 500 scans. If 80% of scans happen in first 2 hours = 400 scans in 7,200 seconds = 0.056 RPS average. Peak burst = 10× average at 30-minute mark = ~0.56 RPS peak. 2× for headroom = ~1.1 RPS. Adjust numbers to coalition's actual mail-drop volume. [ASSUMED — no authoritative Bulgarian SMB campaign conversion data; coalition must confirm volume]

**k6 scenario (constant-arrival-rate pattern):**
```javascript

export const options = {
```

### Anti-Patterns to Avoid

- **Importing `@maxmind/geoip2-node` in middleware.ts:** The Reader.open() calls `fs.readFile`. Node `fs` module is unavailable in Next.js Edge runtime. Worker is the only safe import site. [VERIFIED: Edge runtime constraint, D-08]
- **Importing `@/db` or `@/lib/*` in middleware.ts:** Anti-pattern explicitly documented in `src/middleware.ts` file header. [VERIFIED: codebase]
- **Awaiting the attribution queue enqueue before returning a response:** The middleware must `enqueue(job)` without `await` on the response write path (fire-and-forget), or move the enqueue to a `waitUntil()` call. Use `waitUntil` if available in Next.js Edge: `event.waitUntil(addAttributionJob(payload))`. If not available in the middleware context, use a non-blocking `addAttributionJob(payload).catch(() => {})` pattern.
- **Storing `raw_ip` in any Drizzle column on `attribution_events`:** Violates D-19. Unit test must grep schema file.
- **Setting `attr_sid` cookie in the page response (HTML response):** Triggers Cloudflare's Set-Cookie bypass behavior. Cookie must be set without touching the cached page responses.
- **Using `pg enum` for `self_reported_source`:** Changing a pg enum requires `ALTER TYPE … ADD VALUE` which is DDL and must be explicit in migrations. Using `text` with Zod-enforced enum at API boundary is simpler and forward-compatible (D-11 defers v2 additions). [ASSUMED: team preference for text over pg enum based on existing sector/role pattern in auth.ts]

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| IP → country/region | Custom HTTP call to ipapi.co or ipinfo.io | `@maxmind/geoip2-node` Reader (D-01) | Legal doc mandates GeoLite2; no external HTTP per request; deterministic latency |

## Common Pitfalls

### Pitfall 1: GeoLite2 Reader Loaded in Edge Runtime

**What goes wrong:** `import { Reader } from '@maxmind/geoip2-node'` in `middleware.ts` causes a Next.js build error or silent runtime failure because `fs` and `path` are not available in the Edge runtime.

**Why it happens:** The mmdb reader calls `fs.readFile` synchronously or asynchronously during `Reader.open()`. The Edge runtime strips Node.js built-ins.

**How to avoid:** Import `@maxmind/geoip2-node` only in `src/lib/geoip.ts` and `src/lib/attribution/worker.ts` (Node runtime). Never in `middleware.ts`, never in `'use client'` components.

**Warning signs:** Build output contains `Module not found: Can't resolve 'fs'`; Edge function logs show `TypeError: fs is not defined`.

### Pitfall 2: attr_sid Cookie Causes Cache BYPASS on Landing Page

**What goes wrong:** Setting `attr_sid` cookie in the response to `GET /` causes Cloudflare to return `cf-cache-status: BYPASS` for every landing page request, eliminating PUB-02 CDN caching and causing QR launch traffic to hit origin directly.

**Why it happens:** Cloudflare's default behavior: "If origin cache control is enabled, Cloudflare does not cache the asset and preserves the Set-Cookie" — any response with Set-Cookie header gets bypassed. [VERIFIED: Cloudflare cache docs]

**How to avoid:** Either (A) move cookie setting to `/api/attr/init` endpoint (client-side JS fires once after page load), or (B) accept first-visit MISS for new users. The Phase 2 OPS-RUNBOOK §2 already has a "bypass cache for authenticated members" rule — `attr_sid` must not unintentionally trigger similar bypass for anonymous visitors.

**Warning signs:** `curl -sI https://chastnik.eu/ | grep cf-cache-status` returns `BYPASS` instead of `HIT` on repeat requests.

### Pitfall 3: BullMQ Producer Used in Edge Middleware

**What goes wrong:** BullMQ uses IORedis which uses `net.createConnection` (Node.js TCP socket). Edge runtime does not support TCP sockets. Importing BullMQ in middleware.ts causes a build error.

**Why it happens:** The middleware must remain Edge-runtime safe. BullMQ + IORedis are Node.js-only.

**How to avoid:** The attribution enqueue must happen in a Node.js route handler or Server Action, not in Edge middleware directly. The clean architecture is: middleware extracts UTM/IP/sid data and passes it as query params or a header to a Node.js API route (`/api/attr`), which performs the BullMQ enqueue. Alternatively, use Upstash Redis's HTTP API (`@upstash/redis`) directly in Edge middleware (Upstash supports Edge/Cloudflare Workers). [ASSUMED: `@upstash/redis` HTTP API is Edge-compatible; verify by checking `@upstash/redis` README for Edge runtime support before implementing].

**Warning signs:** Build error `Module not found: Can't resolve 'net'` in middleware bundle; or `Error: The "net" module is not available for this type of module (Edge Runtime)`.

### Pitfall 4: maxmind Subdivision Code Ambiguity

**What goes wrong:** `response.subdivisions[0].isoCode` returns `"22"` (just the numeric suffix, not `"BG-22"`). Dashboard stores "22" but displays it without the country prefix — ambiguous if you ever add non-Bulgarian traffic.

**How to avoid:** Construct the full ISO code in `lookupIp()`: `const subdivCode = response.country.isoCode + '-' + response.subdivisions[0].isoCode`. Store `"BG-22"` in `attribution_events.first_oblast`. Dashboard filter label maps `"BG-22"` → `"София-град"` via a lookup table (28 entries, no DB query needed).

**Warning signs:** Dashboard shows "22" instead of "BG-22" in oblast filter; oblast values lack country prefix.

### Pitfall 5: attribution_events Grows Unbounded on QR Launch

**What goes wrong:** QR launch sends 10,000+ letters; every unique visitor creates one `attribution_events` row; but QR bots and URL scanners (security tools, email preview crawlers) also hit the URL, creating phantom rows.

**Why it happens:** `attr_sid` cookie is set for every request including bot traffic. Bots don't send cookies back on second request, so each bot hit INSERTs a new row instead of UPDATEing an existing one.

**How to avoid:** The one-row-per-session model (D-04) already limits growth to ~1 row per unique browser session. For bots: add `user-agent` heuristic filtering in the worker — if UA matches common bot patterns, write `first_qr_flag = 'bot_suspect'` and skip GeoIP to save compute. This is advisory, not blocking. [ASSUMED: no authoritative bot-UA list; heuristic only]

### Pitfall 6: PII in Pino Logs from Attribution Worker

**What goes wrong:** Worker logs the job payload which includes `raw_ip` for debugging. Pino outputs the full JSON to stdout, which is captured by Fly.io's log aggregator and forwarded to Logtail/Better Stack. Raw IPs appear in centralized logs.

**Why it happens:** Default Pino logger stringifies the entire job object for debugging.

**How to avoid:** Apply the Phase 1 D-21 PII redaction pattern: configure Pino's `redact` option with `['raw_ip', 'ip']` paths. Log only `{ attr_sid, task: 'geoip', oblast, country, lat_ms }` after GeoIP lookup (not before). [VERIFIED: existing Pino redact pattern in Phase 1 `src/lib/logger.ts`]

**Warning signs:** `grep -r "raw_ip\|x-forwarded-for" /var/log/fly` shows IP addresses in log output.

```typescript
// Source: Context7 @maxmind/geoip2-node [VERIFIED]
export async function lookupIp(ip: string): Promise<{ oblast: string; country: string }> {
  try {
    const reader = await getGeoReader();
    const response = reader.city(ip);
    const countryCode = response.country?.isoCode ?? null;
    const subdivSuffix = response.subdivisions?.[0]?.isoCode ?? null;
    const oblast = (countryCode && subdivSuffix)
      ? `${countryCode}-${subdivSuffix}` // e.g. 'BG-22'
      : 'unknown';
    const country = countryCode ?? 'unknown';
    return { oblast, country };
  } catch {
    // D-03: always write row; oblast = 'unknown'
    return { oblast: 'unknown', country: 'unknown' };
  }
}
```

## Security Domain

### Applicable ASVS Categories (Level 1)

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | No (no new auth system) | — |
| V3 Session Management | Partial | `attr_sid` cookie: HttpOnly, SameSite=Lax, Secure, 30-day expiry — matches ASVS 3.4 cookie requirements |
| V4 Access Control | Yes | Dashboard Server Action: role check `['admin', 'editor'].includes(role)` before any Drizzle query |
| V5 Input Validation | Yes | UTM params: sanitized via `text` column type (no SQL injection via Drizzle parameterized queries); `self_reported_source` validated via Zod enum before DB write |
| V6 Cryptography | No | No new crypto; `attr_sid` is UUID v4 (sufficient for session token, not HMAC-required) |
| V8 Data Protection | Yes | No raw IP in DB (D-19 / GDPR-09); Pino redact pattern for worker logs; attribution row cascade-deletes with user |
| V12 File/CDN | Partial | MaxMind .mmdb file integrity: download uses HTTPS; consider SHA-256 checksum verification of downloaded file against MaxMind's published hash |
| V13 API Access | Yes | Dashboard endpoint: role-gated Server Action; not exposed to public internet directly (Payload admin at `/admin/*`) |

### Known Threat Patterns

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| UTM parameter injection (e.g., `utm_source=<script>`) | Tampering | Drizzle parameterized queries prevent SQL injection; values stored as text, rendered escaped in React |
| Raw IP logged in Pino output | Information Disclosure | Pino `redact: ['raw_ip', '*.raw_ip']` in attribution worker logger config |
| Anonymous-to-user linkage forgery (crafting attr_sid to hijack another user's attribution) | Tampering | `attr_sid` is UUID v4 (122 bits of entropy); collision probability negligible; HttpOnly prevents JS read |
| Attribution dashboard accessed by unauthenticated user | Elevation of Privilege | Server Action + RSC role check; Payload admin shell requires admin login to render at all |
| mmdb file tampered in build artifact | Tampering (supply chain) | Download over HTTPS from download.maxmind.com; optionally verify SHA-256 of downloaded file |
| QR scan surge used to exhaust Upstash Redis quota | Denial of Service | Existing rate-limit middleware (Phase 1 D-19); attribution queue jobs are small payloads; Upstash free tier is 10k cmd/day — review and upgrade if QR volume exceeds threshold |

**Note on Upstash free tier:** At 10,000 commands/day free tier, a QR campaign with 5,000 visitors generates ~5,000 `LPUSH` commands + ~5,000 `BRPOP` commands = ~10,000 total. Tight. If mail-drop volume exceeds 5,000 scans/day, upgrade Upstash plan before launch. [ASSUMED: Upstash current free tier is 10k cmd/day; verify at upstash.com/pricing before launch]
