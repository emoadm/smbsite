# Phase 02.1: Attribution + Source Dashboard - Context

**Gathered:** 2026-05-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Deliver source attribution for the public site (must complete before the QR mail drop). Scope:

1. **Server-side attribution capture on every visit** — UTM params (source, medium, campaign, term, content), HTTP Referer, IP-derived oblast (Bulgarian region) + country, QR-flag derivation, landing path. Captured by Edge middleware via a first-party `attr_sid` session cookie. Raw IP is converted in-memory to (oblast, country) and discarded BEFORE the database write.
2. **Anonymous-session-to-user linkage** — pre-registration events keyed by `attr_sid`; on email confirmation the row is UPDATEd with `user_id` (per ATTR-05). Both first-touch (initial) and last-touch (most-recent) attribution columns are persisted.
3. **"Откъде научихте" registration field** — required dropdown with 8 fixed Bulgarian options + an optional `Друго` free-text input that appears only when "Друго" is picked. Stored on the `users` row at registration submission.
4. **MaxMind GeoLite2 in-process geolocation** — `GeoLite2-City.mmdb` auto-downloaded in the Docker build via `MAXMIND_LICENSE_KEY` secret. Zero external HTTP per request, zero new sub-processor exposure for IP data.
5. **Editor-facing source-attribution dashboard inside Payload admin** — custom view at `/admin/views/attribution`. Date-range filter (default: last 7 days), aggregate counts per UTM source / oblast / QR flag / self-reported source, one chart (registrations-by-oblast bar list), CSV export.
6. **Non-blocking attribution writes** — fire-and-forget via existing BullMQ + Upstash queue (Phase 1 D-19) so attribution work never adds latency to the public TTFB.
7. **OPS-05 2x QR-peak load test** — k6 from EU runner against staging/preview deploy, run as the final task before phase ship. Failure blocks the phase ship + the QR mail drop.

**Explicitly NOT in this phase:**
- Per-user referral / "purvani priateli" share-link mechanism — punted to its own future phase to keep Phase 2.1 focused on QR/UTM measurement
- Full BI-lite dashboard (per-day timelines, scan-to-registration funnel, saved filter presets) — defer until warmup volume justifies it
- GDPR DSAR export integration of attribution rows — implemented inside Phase 6 GDPR-04 (this phase ensures the row is cascade-deletable; export-formatting is Phase 6's job)
- Real video content / coalition channels (D-CoalitionChannels) — separate parallel coalition track
- Idea catalog, voting, proposals — Phases 3 & 4

<decisions>
## Implementation Decisions

### GeoIP Source

- **D-01:** GeoIP resolution = **MaxMind GeoLite2 self-hosted (`GeoLite2-City.mmdb`)** queried in-process via the `maxmind` (or `mmdb-lib`) npm package. Zero external HTTP per request, zero new sub-processor exposure, deterministic latency at scale. Matches `attribution-balancing-test.md` as written. Resolves the `attribution-balancing-test.md` ↔ `CLAUDE.md` (ipapi.co) conflict in favor of the legal doc.
- **D-02:** `.mmdb` file = **auto-downloaded in the Docker build** via `MAXMIND_LICENSE_KEY` secret (Fly.io secret + GitHub Actions secret, mirrors Brevo / Sentry secret pattern from Phase 1). MaxMind EULA prohibits redistribution, so the file is never committed to the repo. Refresh cadence = on each deploy (planner may add a scheduled rebuild later if stale-DB drift becomes measurable). Image size bump ~80 MB is acceptable.
- **D-03:** Failed-lookup behavior (private IP, IPv6 miss, lookup error) = **always write the attribution row; oblast = `"unknown"` (string sentinel), country = best-effort or `"unknown"`**. Matches D-19 fire-and-forget spirit; dashboard surfaces `unknown` as its own bucket alongside named oblasts.

### Attribution Storage Model

- **D-04:** Recording grain = **one row per anonymous session** in `attribution_events`. First touch INSERTs the row with `first_*` columns; subsequent visits within the same session UPDATE the `last_*` columns. Bounded volume (≈ one row per visitor), enough fidelity for both first-touch and last-touch dashboards.
- **D-05:** Anonymous-session identity = **first-party cookie `attr_sid`** set by Edge middleware. UUID v4, 30-day expiry, `SameSite=Lax`, `HttpOnly`, `Secure`. Strictly-necessary cookie category (no consent gate — attribution is legitimate-interest per Phase 1 D-14). Survives across pages without per-request DB hits. Cookie set lives in `src/middleware.ts` alongside the existing cf-ray gate (Edge-runtime safe, no `@/lib` or `@/db` imports).
- **D-06:** Attribution columns persisted as **first-touch + last-touch pair**: `first_utm_{source,medium,campaign,term,content}`, `first_referer`, `first_oblast`, `first_country`, `first_qr_flag`, `first_landing_path`, `first_seen_at` AND parallel `last_*` columns. Dashboard answers both "which channel started the funnel?" and "which channel converted?" (industry-standard split-attribution model).
- **D-07:** Anonymous-to-user linkage = **on email confirmation** (per ATTR-05). The `verifyOtp` Server Action reads the request's `attr_sid` cookie, looks up the matching `attribution_events` row, UPDATEs `user_id`. If no `attr_sid` cookie present (rare — direct registration without prior attr cookie), no linkage occurs and the user row carries only the self-reported source.
- **D-08:** Attribution writes are **non-blocking on the request path** — enqueued to existing BullMQ queue (Phase 1 D-19 reuse). The middleware sets the cookie + reads UTM/Referer synchronously (cheap header reads) and enqueues a job carrying `{attr_sid, raw_ip, ua, utm_*, referer, landing_path, ts}`. The queue worker performs the GeoIP lookup, derives oblast, and INSERTs/UPDATEs `attribution_events`. **Raw IP exists only inside the job payload (Redis-resident, ephemeral) and is discarded after the worker completes the lookup.** No raw IP ever reaches Postgres. Worker-level guarantee verified by a unit test that asserts no `inet`/`text` raw-IP columns in `attribution_events` schema.

### "Откъде научихте" Registration Field (ATTR-06)

- **D-09:** Field UX = **required dropdown + optional `Друго` free-text input that appears conditionally when user picks `Друго`**. Mirrors Phase 1 sector/role dropdown pattern (D-09, D-10). Best-of-both: clean analytics buckets + escape hatch for long-tail mentions.
- **D-10:** Locked Bulgarian options (v1):
  1. `QR код в писмо`
  2. `Имейл от коалицията`
  3. `Сайт sinyabulgaria.bg`
  4. `Facebook`
  5. `LinkedIn`
  6. `Препоръка от приятел/колега`
  7. `Новина / медия`
  8. `Друго` (triggers free-text input)

  Sinya Bulgaria official site option included to surface organic spillover from the coalition's main site. Telegram + WhatsApp Channel options deliberately deferred — D-CoalitionChannels haven't launched yet so they would be 0% at warmup time; revisit post-warmup.
- **D-11:** Schema placement = **on the `users` row**. New columns: `self_reported_source` (text or pg enum) + `self_reported_other` (nullable text, populated only when `self_reported_source = 'other'`). Captured at registration submission, never mutated. Cascade-deletes naturally with the user row (Phase 6 GDPR-05 forward-prep). Internal enum values use English snake_case (`qr_letter`, `email_coalition`, `sinya_site`, `facebook`, `linkedin`, `referral`, `news_media`, `other`); display labels are next-intl keys.

### Source-Attribution Dashboard (ATTR-07)

- **D-12:** Dashboard scope (v1) = **tables + one key chart + filters + CSV export**, no per-day timeline / no funnel / no saved presets. Filters: date range (default last 7 days), UTM source, oblast, QR flag, self-reported source. Aggregate counts per UTM source / oblast / QR flag / self-reported source. One chart: registrations-by-oblast as a bar list (no Bulgaria-map SVG; defer until volume justifies). CSV export of the filtered rows. Lives as a custom Payload admin view at `/admin/views/attribution` (Payload 3 supports custom React views inside the admin shell).
- **D-13:** Dashboard data source = **read-only Drizzle queries** against `attribution_events` + `users`. No direct Payload collection wrapper for `attribution_events` (we control the schema via Drizzle migrations, not Payload field definitions; Payload custom view is just a React component fetching aggregates via a Server Action with `editor`-role gate). Avoids fragmenting schema management between Drizzle and Payload.

### Referral Mechanism (Phase 2 Deferred Item)

- **D-14:** Phase 2's deferred "purvani priateli" referral mechanism = **PUNTED to its own future phase** (post-warmup). Per-user share-link generation, member dashboard share page, and `referrer_id` on `users` are out of scope here. Rationale: Phase 2.1's gating purpose is QR/UTM measurement before the mail drop; referral is conceptually distinct (organic word-of-mouth) and adds substantial UX scope (member dashboard, share-page). Add as `D-ReferralMechanism` deferred item in STATE.md.

### Load Test (OPS-05)

- **D-15:** Load test tool = **k6**. Scriptable JS, JSON output for CI artifacting, actively maintained in 2026. Run from a one-off Hetzner EU VM (cheap, EU egress) targeting the **staging / preview deploy** (not prod) so the test exercises real Cloudflare → Fly.io path without affecting live traffic.
- **D-16:** Load test scenario = **2x expected QR-peak**. Planner derives the target RPS from the coalition's mail-drop volume × expected scan-conversion-window. Scenarios MUST exercise: anonymous landing-page hit (cookie set + UTM capture path), `attribution_events` write throughput, registration form submission with `self_reported_source` field, OTP verify path (linkage UPDATE).
- **D-17:** Load test timing = **final task of Phase 2.1**, run after all attribution code paths exist and Cloudflare cache rules are set. Failure = phase-ship blocker AND QR mail drop blocker. Output committed to `.planning/phases/02.1-attribution-source-dashboard/02.1-LOAD-TEST.md` (k6 JSON summary + p50/p95/p99 latencies + error rate + observations).

### Carry-forward Locks (Not Re-litigated Here)

### Post-Research Decisions (locked 2026-05-03)

- **D-22 (resolves D-CookieVaryCacheRule + Open Q1):** Cookie-set strategy = **Option A — dedicated `/api/attr/init` Node-runtime endpoint**. Edge middleware does NOT set `attr_sid` on landing-page responses; landing pages remain anonymously cacheable on Cloudflare. Client-side JS on landing pages fires `fetch('/api/attr/init', { credentials: 'include' })` once on first load. The `/api/attr/init` endpoint: (a) sets the `attr_sid` cookie, (b) reads UTM params + Referer from the request, (c) extracts raw IP via `getClientIp` from `src/lib/ip.ts`, (d) enqueues a BullMQ attribution job via the existing `src/lib/email/queue.ts` producer pattern. Rationale: QR mail drop = bursty new visitors; cache MISS amplification on landing pages would be a launch-day risk. One extra round-trip on first visit is acceptable.
- **D-23 (resolves Open Q2):** Edge → BullMQ enqueue path = **Node.js intermediary via `/api/attr/init`**. No direct `@upstash/redis` HTTP push from Edge runtime; the existing IORedis-based BullMQ producer is reused as-is from `src/lib/email/queue.ts`. Combines cleanly with D-22 (the same `/api/attr/init` endpoint owns both cookie set and enqueue). Edge middleware retains its cf-ray gate role only — no attribution responsibilities, no BullMQ imports. Eliminates Edge-runtime compatibility risk for IORedis.
- **D-24 (resolves Open Q3 from RESEARCH.md):** Oblast storage format = **ISO 3166-2 code only** (`first_oblast = 'BG-22'`, `last_oblast = 'BG-22'`). Bulgarian display names mapped via a static 28-entry constant in the dashboard component. Keeps DB lean; planner adds a `lib/oblast-names.ts` constant lookup.
- **D-25 (resolves Open Q4 from RESEARCH.md):** CLAUDE.md `ipapi.co` row = **add inline footnote** noting "Override: Phase 2.1 D-01 uses MaxMind GeoLite2 (see legal/attribution-balancing-test.md)". Do NOT delete the row — retain for hypothetical future lighter-weight lookup needs.
- **D-26:** k6 target RPS derivation — `mail_drop_volume × scan_rate_pct × peak_compression_factor / peak_window_seconds` (formula in RESEARCH.md § Validation Architecture). Default `TARGET_RPS=2` env var with derivation table committed to `02.1-LOAD-TEST.md`. Coalition can override before launch by re-running `TARGET_RPS=N k6 run attribution-load.js`. No user blocker.

### Claude's Discretion
