## Why this test gates phase ship + QR mail drop

D-17: failure of OPS-05 blocks both Phase 2.1 ship AND the QR mail drop. The
QR campaign's launch-day burst is the highest-load moment the platform will
ever see; if the attribution + registration + OTP-verify path can't sustain 2×
the projected peak with comfortable latency headroom, the user-visible result
will be slow form submissions, lost attribution data, broken email-confirmation
linkage, and a credibility hit for the coalition on day zero.

D-16 explicitly mandates the OTP verify path is exercised under load — that's
the moment `attribution_events.user_id` linkage UPDATE fires (Plan 06 Task 2).
A fast register POST means nothing if the OTP-verify path serializes badly.

## 4. Pre-run staging seed (D-16 OTP leg)

The `otp_verify_hit` k6 scenario submits a fixed OTP code against a seeded
test user. This MUST be done on staging only — production seed is intentionally
absent so the load fixture cannot be replayed against real accounts.

Connect to the staging DB (Neon staging branch) and run:

```sql
-- !!! STAGING DB ONLY — DO NOT RUN THIS AGAINST PRODUCTION !!!
-- Seeds the load-test fixture user + a long-lived OTP verification token.
-- The k6 otp_verify_hit scenario will hammer this user repeatedly.

-- 1. Insert (or upsert) the fixture user
INSERT INTO users (email, name, full_name, sector, role, self_reported_source, "emailVerified")
VALUES ('loadtest+otp@chastnik.eu', 'Load Test', 'Load Test', 'services', 'owner', 'qr_letter', NULL)
ON CONFLICT (email) DO UPDATE SET "emailVerified" = NULL, full_name = EXCLUDED.full_name;

-- 2. Compute the HMAC of code '000000' for this email using the same
--    hashOtp() pipeline used by request-otp.ts. Easiest path: invoke the
--    helper from a one-off staging script:
--      pnpm tsx scripts/seed-loadtest-otp.ts loadtest+otp@chastnik.eu 000000
--    (operator: write this 5-line helper if it doesn't exist; it should
--     import hashOtp from '@/lib/auth-utils' and INSERT into verificationTokens
--     with expires = now() + interval '7 days' and attempts = '0'.)
--
-- 3. Verify the token is in place:
SELECT identifier, expires, attempts FROM "verificationTokens"
WHERE identifier = 'loadtest+otp@chastnik.eu';
```

Alternative: if Phase 1 added a feature-flagged "test OTP" mode that accepts
a known constant when `STAGING_TEST_OTP_BYPASS=true`, set that flag instead
and skip the verificationTokens INSERT. Check `src/app/actions/verify-otp.ts`
for any test-mode handling before authoring the seed script.

After the load run completes, clean up:

```sql
DELETE FROM "verificationTokens" WHERE identifier = 'loadtest+otp@chastnik.eu';
DELETE FROM users WHERE email = 'loadtest+otp@chastnik.eu';
```

## Failure path

If any threshold goes RED:

1. STOP the run; do NOT ship Phase 2.1.
2. Investigate the failing leg — Sentry + Better Stack logs from the staging run.
3. Common failure modes:
   - **landing_hit p(95) > 300ms:** Cloudflare cache miss amplification — verify cache rules + cookie strategy (D-22).
   - **attr_init p(95) > 300ms:** Upstash quota exhaustion or BullMQ producer slow — check Upstash command rate; possibly upgrade plan.
   - **register failed > 1%:** Server Action error path — check Sentry; likely Drizzle constraint violation or Turnstile sitekey mismatch.
   - **otp_verify failed > 1% or p(95) > 800ms:** likely (a) `Next-Action` header wrong/missing — re-capture from DevTools; (b) verificationTokens fixture expired or attempts exhausted — re-seed; (c) attr_sid cookie not in jar — verify the prime-the-jar landing+init calls in `otpVerifyHit` are succeeding.
   - **D-16 linked_rows = 0:** linkage UPDATE never fired under load — verify Plan 06 Task 2 shipped + cookie jar is shared across legs in the script.
4. Fix → re-run → re-sign.
