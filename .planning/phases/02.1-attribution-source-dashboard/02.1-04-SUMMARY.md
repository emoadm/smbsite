# Phase 02.1 Plan 04: Attribution BullMQ Producer + Worker + Pino Redact Summary

BullMQ attribution producer + worker pair built as a near-verbatim mirror of the Phase 1 email/queue.ts and email/worker.tsx, with the worker's processor doing in-memory MaxMind lookup, raw IP discard, qr_flag derivation from `utm_medium=qr`, and Drizzle upsert via `onConflictDoUpdate` on `attribution_events.attr_sid`. `scripts/start-worker.ts` now launches both workers in one Node process group (Fly.io worker process group runs both). Pino REDACT extended with `'raw_ip'` as belt-and-braces.

## Final Decisions

- **Retry policy: attempts: 3** — Email retries 5 (user-visible OTP must arrive). Attribution failure means a single visit's row is missing from the dashboard — non-fatal for the QR campaign attribution funnel. Trade-off: less Redis quota burned per failed visit; OPS-RUNBOOK already plans for Upstash quota upgrade.
- **Concurrency: 5** — In-memory mmdb lookup is sub-millisecond CPU. The bottleneck is the Drizzle write; with 5 concurrent jobs we saturate the Neon connection pool but stay under the 20-conn pool cap (Phase 1 default).
- **REDACT diff** — single addition: `'raw_ip'`. No other PII fields changed.
- **start-worker.ts before/after** — Phase 1 single email worker → Phase 2.1 tuple-array `[[emailWorker, 'email'], [attrWorker, 'attribution']]` with `Promise.all(workers.map(([w]) => w.close()))` shutdown.
- **Job name** — `'attribution-event'` (single string) chosen over per-payload variants. Each Redis Insight row reads cleanly: `attribution-queue/attribution-event/<id>`.
