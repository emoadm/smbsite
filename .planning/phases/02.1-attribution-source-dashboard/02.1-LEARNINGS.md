# Phase 02.1 Learnings: attribution-source-dashboard

## Decisions

### MaxMind GeoLite2 self-hosted over ipapi.co (D-01)
GeoIP resolution moved entirely to self-hosted MaxMind GeoLite2 (`@maxmind/geoip2-node@6.3.4`) baked into the runner image; raw IP never leaves the EU container. CLAUDE.md still names ipapi.co for documentation, but Phase 2.1 footnotes (D-25) supersede it for political-platform use.

**Rationale:** Resolves the conflict between `attribution-balancing-test.md` (legal doc, names GeoLite2) and CLAUDE.md (stack guide, names ipapi.co) in favor of the legal doc — raw IP must never cross the EU boundary for a political platform.
**Source:** 02.1-DISCUSSION-LOG.md (Q1), 02.1-02-SUMMARY.md, CONTEXT.md D-01/D-25

---

### Auto-download .mmdb in Docker build via BuildKit secret mount
Dockerfile builder stage uses `RUN --mount=type=secret,id=MAXMIND_LICENSE_KEY` to download `GeoLite2-City.mmdb` fresh on every CI deploy; runner stage `COPY --from=builder` brings the file in. The license key never enters an image layer.

**Rationale:** MaxMind EULA forbids redistribution (rules out vendoring). `geoip-lite` bundled DB has weaker BG oblast precision. Mirrors Brevo / Sentry secret pattern from Phase 1.
**Source:** 02.1-02-SUMMARY.md, 02.1-DISCUSSION-LOG.md (Q2)

---

### Always-write attribution rows with `oblast='unknown'` fallback (D-03)
`lookupIp(ip)` is wrapped so any internal throw (invalid IP, missing mmdb, IPv6 miss, malformed lookup) returns `{ oblast: 'unknown', country: 'unknown' }`. The downstream worker can ALWAYS write the attribution row.

**Rationale:** Dropping the event on lookup failure would lose 10–30% of visits (VPN/IPv6) — wrong for campaign attribution. Dashboard surfaces "Неизвестен" alongside named oblasts.
**Source:** 02.1-02-SUMMARY.md, 02.1-DISCUSSION-LOG.md (Q3), src/lib/geoip.ts

---

### One row per anonymous session with first_*/last_* split-attribution columns (D-04, D-06)
`attribution_events` is one row per `attr_sid` (DB-level UNIQUE). Subsequent visits UPDATE the `last_*` columns; `first_*` columns are immutable after first INSERT. Schema carries 11 first_* + 11 last_* + 3 identity = 25 columns.

**Rationale:** Industry-standard split-attribution preserves both the causal first-touch story and the converting last-touch signal without page-view-stream volume. One row per visit would balloon on QR-peak day.
**Source:** 02.1-01-SUMMARY.md, 02.1-DISCUSSION-LOG.md (Attribution storage shape Q1/Q3), src/db/schema/attribution.ts

---

### First-party `attr_sid` cookie (UUID v4, 30d, HttpOnly, SameSite=Lax) (D-05, D-22)
Anonymous session id is a UUID v4 first-party cookie set ONLY by `/api/attr/init`. Public landing pages remain `Set-Cookie`-free so Cloudflare CDN cache stays warm. SameSite=Lax (not Strict) so QR scans arriving from external referers still set the cookie.

**Rationale:** Server-derived hash of IP+UA breaks linkage when IP rotates. localStorage UUID fails on JS-disabled visits. The Cloudflare-cache invariant is non-negotiable for PUB-02.
**Source:** 02.1-05-SUMMARY.md, 02.1-DISCUSSION-LOG.md (Attribution storage shape Q2), src/app/api/attr/init/route.ts

---

### Required dropdown + optional Други free text on registration (D-09/D-10)
`auth.register.source` exposes 8 fixed enum values + a conditional `other` free-text input (max 300 chars, optional). The `Друго` input renders only when `sourceValue === 'other'`.

**Rationale:** Free-text-only produces messy analytics ("facebook" / "FB" / "фейсбук" all separate). Optional dropdown loses required-field signal. Required + Други is the goldilocks position.
**Source:** 02.1-DISCUSSION-LOG.md ("Where did you hear" UX Q1), 02.1-03-SUMMARY.md, 02.1-06-SUMMARY.md

---

### Edge middleware MUST NOT import `@/lib/attribution` (D-23)
`src/middleware.ts` runs on the Edge runtime (no `fs`, no Node APIs). The matcher already excludes `/api/.*`, so `/api/attr/init` runs as a regular Node API route. Negative grep test asserts middleware contains zero `attr_sid`, zero `addAttributionJob`, zero `from '@/lib/attribution'`.

**Rationale:** `Reader.open()` calls `fs.readFile`, unavailable in Edge — the build would fail with `Module not found: Can't resolve 'fs'`. Keeping middleware free of attribution imports also preserves its sole responsibility (cf-ray gate).
**Source:** 02.1-05-SUMMARY.md, src/lib/geoip.ts (header comment), tests/integration/attr-init-route.test.ts

---

### `/api/attr/init` owns BOTH the cookie set and the BullMQ enqueue
The dedicated Node-runtime endpoint is the only place where `attr_sid` is set and the only place where `addAttributionJob()` is called from request context. `<AttrInit/>` client component fires this endpoint exactly once per browser session via `useEffect` with empty deps.

**Rationale:** Concentrates the Edge-incompatible code path behind a single Node-runtime boundary. Public landing pages stay CDN-cacheable. Client-side instrumentation triggers the server cookie set without invalidating cached HTML.
**Source:** 02.1-05-SUMMARY.md, src/app/api/attr/init/route.ts, src/components/attribution/AttrInit.tsx

---

### Worker `attempts: 3` (vs email's 5)
Attribution worker uses BullMQ `attempts: 3`, half the email worker's retry budget.

**Rationale:** Email retries 5 because OTP must arrive (user-visible). Attribution failure means a single visit's row is missing from the dashboard — non-fatal for the QR campaign attribution funnel. Less Redis quota burned per failed visit.
**Source:** 02.1-04-SUMMARY.md, src/lib/attribution/queue.ts

---

### BullMQ worker concurrency 5
Single Node process runs the attribution worker with `concurrency: 5`.

**Rationale:** In-memory mmdb lookup is sub-millisecond CPU. Bottleneck is the Drizzle write; 5 concurrent jobs saturate the Neon connection pool but stay under the 20-conn pool cap (Phase 1 default).
**Source:** 02.1-04-SUMMARY.md

---

### Pino REDACT extended with `'raw_ip'` (D-19 belt-and-braces)
`src/lib/logger.ts` REDACT array gains `'raw_ip'` even though the worker `.values({})` block has zero raw_ip refs and the schema has zero raw-IP columns.

**Rationale:** Belt-and-braces — three independent layers (schema grep test, worker `.values` grep test, Pino REDACT) so a single regression cannot leak raw IP. Logger fix is a one-line, zero-cost guard.
**Source:** 02.1-04-SUMMARY.md, src/lib/logger.ts:11

---

### Direct `messages/bg.json` import in Payload admin shell
Payload custom admin views (`AttributionView.tsx`, `AttributionDashboard.tsx`) import `messages/bg.json` as a plain object and read `attribution.dashboard.*` directly — no `useTranslations()`.

**Rationale:** Payload admin shell does not provide a next-intl Context. The existing Phase 1 pattern in `src/lib/email/worker.tsx` (also outside the provider) uses the same approach. Trade-off: TypeScript can't enforce key existence (type assertion used), but runtime is fast and there's no provider plumbing.
**Source:** 02.1-07-SUMMARY.md, src/app/(payload)/admin/views/attribution/AttributionDashboard.tsx

---

### Singular `role` field (NOT plural `roles`) is canonical
`admin_users.role` is a Payload `select` field with options `[admin, editor]`. PATTERNS.md / RESEARCH.md mention of plural `roles` was incorrect for this codebase; tests enforce the singular form with a negative grep.

**Rationale:** Source of truth is `src/collections/Users.ts:22-30`. A plural-form drift would silently break the role gate.
**Source:** 02.1-07-SUMMARY.md, src/collections/Users.ts, tests/unit/dashboard-role-gate.test.ts

---

### Dual role gate (RSC + Server Action) for ASVS V4 defense in depth
Role check runs at BOTH the page-component layer (`AttributionView.tsx:49-55`, before `fetchAttributionAggregates()`) AND the Server Action layer (`actions.ts:33-41`, `assertEditorOrAdmin()` re-validates on every call).

**Rationale:** ASVS V4 defense in depth. A future refactor that bypasses the page component (e.g. exposing the Server Action as a public API) can't accidentally leak the dashboard data.
**Source:** 02.1-07-SUMMARY.md, 02.1-SECURITY.md (T-02.1-26, T-02.1-27)

---

### CSV export uses RFC 4180 quoting
`escape(v)` quotes values containing `[",\n]`, doubles embedded `"` to `""`. Filename: `attribution-{dateFromYYYY-MM-DD}-{dateToYYYY-MM-DD}.csv`. Order: `ORDER BY first_seen_at` ASC.

**Rationale:** Spreadsheet-tool compatibility (Excel, Google Sheets, Numbers all parse RFC 4180). Avoids ad-hoc escaping bugs.
**Source:** 02.1-07-SUMMARY.md, src/app/(payload)/admin/views/attribution/actions.ts:122-133

### `db:generate` (committed migration) over `db:push` for worktree executors
Plan 01 generated a committed SQL migration (`0001_grey_umar.sql` + snapshot/journal updates) instead of pushing to Neon directly. CI deploy.yml's existing migrate job applies on next push to main.

**Rationale:** Worktree executor has no `DIRECT_URL` env var (Phase 1 OPS-RUNBOOK keeps it inside Fly + GitHub Actions secrets only). The plan's `<action>` block explicitly accepts this fallback. Same end state, just sequenced behind CI.
**Source:** 02.1-01-SUMMARY.md (Deviations from Plan §Rule 3)

## Lessons

### `node:20-alpine` ships without curl
The base image used for the runner does not include curl. The MaxMind download step needs it just-in-time on the builder stage.

**Context:** Discovered when the secret-mount RUN block needed `curl -L download.maxmind.com/...`. Resolved by adding `RUN apk add --no-cache curl` to the builder stage unconditionally (precedes the secret-mount RUN). Runner stage NOT modified — runner does not need curl at runtime.
**Source:** 02.1-02-SUMMARY.md (Curl-Availability Decision)

---

### MaxMind `Reader.open()` returns `ReaderModel`, not `Reader`
`Reader` in `@maxmind/geoip2-node@6.3.4` is the static factory class; `Reader.open()` returns a `ReaderModel` instance, which is the type that exposes the `.city(ip)` method.

**Context:** First `tsc --noEmit` after the GREEN commit reported `error TS2339: Property 'city' does not exist on type 'Reader'`. The plan's example code (and RESEARCH.md Pattern 1 snippet) typed the singleton as `Reader | null`. Fix: import `ReaderModel` as a type and change the singleton + return type.
**Source:** 02.1-02-SUMMARY.md (Auto-fixed Issues §Rule 1)

---

### busybox tar (Alpine) chokes on `--wildcards`
The Alpine-bundled busybox `tar` cannot extract specific patterns from a tarball using `--strip-components --wildcards "*/GeoLite2-City.mmdb"`. Builder stage now installs GNU tar.

**Context:** Surfaced during operator's staging Docker build (commit `268fee6`). MaxMind ships GeoLite2-City as a tar.gz; busybox tar's wildcard handling is incompatible. Fix: `apk add --no-cache tar` and download size verified before extraction.
**Source:** 02.1-08-SUMMARY.md (commit 268fee6)

---

### Build-time env vars matter: Auth.js + Drizzle init at module-eval time
`EMAIL_FROM_TRANSACTIONAL` and `DATABASE_URL` must be present at build time, not just runtime. Auth.js email provider configuration and the Drizzle client routing branch both evaluate at module load — missing env vars throw at builder stage, not at first request.

**Context:** Two prod-CI bugs surfaced during operator's staging build attempts (commits `77666cf`, `b4b03c8`). Fix: GitHub Actions `secrets:` block in `deploy.yml` now passes both as `--build-arg` so they exist at `pnpm build` time.
**Source:** 02.1-08-SUMMARY.md (operator-surfaced bugs table)

---

### `tsconfig.json` must be in the runner image for tsx alias resolution
`scripts/start-worker.ts` runs via `node --import tsx` in the runner; tsx resolves `@/*` path aliases by reading `tsconfig.json`. Runner image needed an explicit `COPY tsconfig.json` (commit `61dbfd5`).

**Context:** Worker process couldn't import `@/lib/attribution/worker` at runtime — module resolution failed because the runner image previously copied only `package.json` and built artifacts. Surfaced during staging deploy.
**Source:** 02.1-08-SUMMARY.md (commit 61dbfd5)

---

### Payload's importMap + sibling `page.tsx` create shadow routing
Payload 3.84's catchall route + importMap registration is sufficient to route `/admin/views/attribution`. Adding a sibling `page.tsx` next to the registered Component file makes Next.js rules force a default export AND hijacks the route from the importMap-resolved view.

**Context:** Plan 07 originally placed `AttributionView` inside `page.tsx` with a default export. Operator commit `b5aa721` extracted it to `AttributionView.tsx` and updated `payload.config.ts` Component path to `AttributionView#AttributionView` after observing the page render blank. The drop-shadowing fix unblocked the dashboard.
**Source:** 02.1-08-SUMMARY.md (commit b5aa721), 02.1-VERIFICATION.md (lines 224-238)

---

### Plain form-encoded POSTs do NOT dispatch React Server Actions
A `http.post('/auth/otp', form)` from k6 returned 404 — Server Actions require an RSC multipart envelope with a per-iteration boundary. The action ID lives in headers, not in the URL.

**Context:** k6 OTP leg failed to invoke `verifyOtp`. Operator added `buildOtpVerifyBody()` that constructs the RSC multipart wire format (commit `f337f88`). Without this, the linkage UPDATE never fires and D-16 cannot be measured.
**Source:** 02.1-08-SUMMARY.md (commit f337f88), 02.1-LOAD-TEST.md (run #1 observations)

---

### Test Turnstile sitekey alone isn't enough to bypass CAPTCHA gate
The original `isTestBuild` guard checked only the sitekey. Staging deploys use the always-pass test sitekey too (k6 cannot solve real CAPTCHA), so the previous guard short-circuited the entire enqueue path on staging — D-16 linkage was untestable.

**Context:** k6 run #2 reported `linked_rows = 0` despite green checks. Operator fix (commit `f81f21a`): guard requires sitekey AND `NODE_ENV === 'test'` so deployed staging exercises real Redis but still bypasses Turnstile.
**Source:** 02.1-08-SUMMARY.md (commit f81f21a)

---

### Payload `importMap.js` stays empty until Component paths are registered
The Phase-1-generated `src/app/(payload)/admin/importMap.js` was an empty stub. Custom views need an explicit entry — Payload does not auto-populate it from `payload.config.ts`.

**Context:** Dashboard rendered as a blank white page after the first deploy that registered the custom view. Fix in commit `c37a09b` populated the importMap with the AttributionView entry. UAT Test 10 was blocked until this landed.
**Source:** 02.1-UAT.md (Gaps §1, resolved_by_commits c37a09b)

---

### Worker memory of 256MB OOMs under real load
Production worker process group OOM'd while draining the BullMQ queue under k6 burst. Bumped to 1024MB in `fly.toml` (commit `31236d0`).

**Context:** k6 run #2 observed worker queue backup > 5s under the 4-leg scenario; investigation showed the worker process being killed by Fly's memory enforcement. Run #3 GREEN after the bump.
**Source:** 02.1-08-SUMMARY.md (final run results §Worker queue backup), commit 31236d0

### Node 20 + `undici@7` CacheStorage is incompatible
CI runner Node 20 fails to load `undici@7`'s CacheStorage symbols (added in newer Node V8). Bumped runner Node to 22 in deploy.yml (commit `fe4c261`).

**Context:** CI deploy job started failing during dependency install after a transitive bump. Forward-prep — the same constraint applies to Phase 5 newsletter work (also undici-dependent).
**Source:** 02.1-UAT.md (Gaps §1 artifacts), commit fe4c261

---

### `payload migrate` CLI broken on payload@3.84 + tsx@4.21 + Node 22 ESM
Running `pnpm payload migrate` errors at module load — a known incompat in the patch-package todo list. CI deploy.yml's `payload migrate` step is currently commented out (commit `d603fc8`); Drizzle migrations apply via `drizzle-kit migrate` instead.

**Context:** Surfaced during pre-Phase-2.1 ops; operator deferred the patch-package fix. Current path: Drizzle owns schema migrations; Payload's own migrations table is unused (Payload writes its admin schema via auto-DDL on boot). Tracked todo: `.planning/todos/pending/2026-05-04-payload-tsx-esm-incompat.md`.
**Source:** 02.1-UAT.md (Gaps §1 followup_todos), commit d603fc8

## Patterns

### Forbidden-token schema grep test
For any schema column that should never exist (raw IP, plaintext password, social-security-number-shaped column), readFileSync the schema source, strip `^//` comment lines, then `expect.not.toMatch` on the banned token regex. The schema header may legitimately document WHY the columns are banned — comment-stripping prevents the test going RED on its own documentation.

**When to use:** Any GDPR/PII boundary that lives at the schema layer. Critical invariants (cascade FK, UNIQUE constraint, PII-absence) are asserted at code-review time, not runtime — regression-prevention-as-source-grep.
**Source:** 02.1-01-SUMMARY.md (patterns-established), tests/unit/attribution-schema.test.ts

---

### Lazy module-local singleton (mirror IORedis pattern)
Module-local `_reader: ReaderModel | null = null`; `getGeoReader()` returns the singleton, opening it on first call. `__resetForTests()` exported solely so unit tests can swap config between cases without process-level singleton leak.

**When to use:** Any expensive resource (DB reader, Redis connection, MaxMind .mmdb file) consumed by Node-runtime code. Mirrors `src/lib/email/queue.ts` lines 16-27.
**Source:** 02.1-02-SUMMARY.md (tech-stack patterns), src/lib/geoip.ts

---

### BuildKit secret-mount for build-time secrets
`RUN --mount=type=secret,id=NAME ...` reads the secret from `/run/secrets/NAME` during a single RUN, never persisting it to an image layer. Verified via `docker history --no-trunc <image>` finding zero secret bytes.

**When to use:** API keys / license keys needed only at build time (downloads, signing, code-signing). Pair with Fly's `--build-secret` flag and a matching GitHub Actions secret.
**Source:** 02.1-02-SUMMARY.md, Dockerfile:33-50

---

### Tone-lock vitest pattern
`JSON.stringify(namespace)` then `.not.toMatch(/Уважаеми/)` and `.not.toMatch(/Уважаема/)` — cheap structural check that no vocative form has crept into the formal-respectful Bulgarian copy.

**When to use:** Any new bg.json namespace. One assertion per banned form. Mirrors `tests/unit/queue.test.ts:35-37` (Phase 1 D-27 check).
**Source:** 02.1-03-SUMMARY.md (patterns-established), tests/unit/attribution-i18n.test.ts

---

### Greppable enum lock-in test
For each new Zod enum that appears in a Server Action, add a vitest source-grep test that reads `messages/bg.json` and asserts each enum value resolves to a non-empty Bulgarian string. Drift between the Zod array and bg.json keys turns the lock-in test red.

**When to use:** Any registration/onboarding form that pairs a Zod enum with a Bulgarian dropdown. Prevents the "user reports their selected option doesn't exist anymore" silent regression.
**Source:** 02.1-03-SUMMARY.md (patterns-established), tests/unit/attribution-i18n.test.ts

---

### Drizzle nullable user FK with cascade
`uuid('user_id').references(() => users.id, { onDelete: 'cascade' })` — no `.notNull()`. One declaration covers both deferred linkage (anonymous attribution before registration) AND GDPR cascade (delete attribution when user deletes account).

**When to use:** Any forward-prep "user-attached-later" table where the relationship is optional pre-registration but the FK must cascade for GDPR Article 17.
**Source:** 02.1-01-SUMMARY.md (patterns-established), src/db/schema/attribution.ts:31

---

### Drizzle index naming convention
`<table_short>_<col>_idx`, declared inside the `(t) => ({ ... })` builder. Five indexes per `attribution_events` (sid, user, first_seen, utm_source, oblast).

**When to use:** Any new Drizzle table. Extending Phase 1 `consents.ts` convention. Predictable names make Postgres `\d` output and Sentry slow-query traces self-documenting.
**Source:** 02.1-01-SUMMARY.md (patterns-established), src/db/schema/attribution.ts

---

### Fire-and-forget enqueue
`void addJob(...).catch(() => {})` — no await on the response path. Route returns 200 immediately; job dispatch happens in the background.

**When to use:** Any non-user-visible side-effect (analytics, attribution, audit log). The visitor should not pay the queue-write latency on TTFB. Pair with `Cache-Control: no-store` so the empty-success response isn't cached.
**Source:** 02.1-05-SUMMARY.md (tech-stack patterns), src/app/api/attr/init/route.ts:71-85

---

### Per-leg k6 thresholds (not just global)
k6 scenario declares per-leg `http_req_duration{scenario:landing_hit}: ['p(95)<300']` etc. in addition to the global `p(95)<500`. A slow OTP leg can no longer hide behind a fast landing leg's average.

**When to use:** Any multi-leg funnel load test where one leg's degradation is a real risk (DB write, external API call, expensive query). Tighter per-leg budgets surface bottlenecks before global p(95) does.
**Source:** 02.1-08-SUMMARY.md (key-decisions), .planning/phases/02.1-attribution-source-dashboard/k6/attribution-load.js:106-112

---

### RSC multipart wire format builder for k6 → Server Action POSTs
`buildOtpVerifyBody()` constructs the multipart envelope with a per-iteration boundary that React Server Actions require. Plain form-encoded POSTs hit the route but don't dispatch the action.

**When to use:** Any load test that must exercise a Server Action (not just a public route). One-time investment per project; reusable across phases.
**Source:** 02.1-08-SUMMARY.md (operator addition), .planning/phases/02.1-attribution-source-dashboard/k6/attribution-load.js

---

### Drizzle `onConflictDoUpdate` for upsert with split-attribution
`db.insert(attribution_events).values({...all cols}).onConflictDoUpdate({ target: attribution_events.attr_sid, set: {...lastColsOnly} })`. First INSERT writes both first_* and last_* identically; subsequent visits update only the last_* columns.

**When to use:** Any one-row-per-key table where re-touches should refresh "current state" but preserve "first observed" history. Cleaner than SELECT-then-INSERT-or-UPDATE round-trip.
**Source:** 02.1-04-SUMMARY.md (tech-stack patterns), src/lib/attribution/worker.ts

---

### Schema-as-enforcement-boundary
Critical invariants (cascade FK exists, UNIQUE constraint exists, no PII columns, no `.notNull()` on optional FKs, barrel export wired) are asserted by file-grep tests on the schema source — not by runtime DB checks. Regression goes RED at `pnpm test:unit` before merge, not at staging deploy.

**When to use:** Any schema invariant that has a hard reason (legal, compliance, data-flow) for never being broken. Cheaper than runtime checks; runs in CI on every PR.
**Source:** 02.1-01-SUMMARY.md (patterns), tests/unit/attribution-schema.test.ts

## Surprises

### Wave 2 parallel subagents stalled at 600s stream-watchdog
Two background-subagent attempts for plan 04 stalled with `Agent stalled: no progress for 600s (stream watchdog did not recover)`. Plans 04, 06, 07 all fell back to inline sequential execution.

**Impact:** No measurable schedule slip — the inline path completed in 25 min for plan 04, comparable to the subagent target. But the parallelization assumption baked into Wave 2 of `gsd-execute-phase` did not hold. Pattern map for future phases: have a fallback runbook ready, don't depend on subagent parallelism for critical-path plans.
**Source:** 02.1-04-SUMMARY.md (Execution Mode), 02.1-06-SUMMARY.md, 02.1-07-SUMMARY.md

---

### Schema column count: plan said 23, explicit list summed to 25
Plan 01's `must_haves` / `done` blurbs said "23 columns" but its `<behavior>` column list explicitly enumerated 3 identity + 11 first_* + 11 last_* = 25. Followed the explicit list; verify regex `^(2[3-9]|[3-9][0-9])$` accepted 25 either way.

**Impact:** Caught at execution; no rework. Highlights that natural-language counts in frontmatter can drift from explicit lists. Future planner: trust the explicit enumeration over the count summary.
**Source:** 02.1-01-SUMMARY.md (Decisions Made §Schema column count)

---

### Four separate build-time env vars failed at the builder stage
Auth.js email provider, Drizzle client routing branch, MaxMind license key, and EMAIL_FROM_TRANSACTIONAL all failed at `docker build` time — not at first request. Operator's first staging deploy surfaced 4 cascading build failures.

**Impact:** Phase 02.1 staging took 3 deploy attempts before coming up. Library init code that runs at module-eval time (not request time) needs build-time env vars even if the runtime path is conditional. Future phases: audit module-eval imports of new libraries for hidden env-var dependencies.
**Source:** 02.1-08-SUMMARY.md (operator-surfaced bugs table), commits 77666cf, b4b03c8, 268fee6, f337f88

---

### k6 D-16 `linked_rows = 1`, not `~150` — fixture-limited
The seed fixture provides exactly one `verificationToken` row that gets consumed on the first verify; subsequent verifies return `auth.otp.expired` (HTTP 200, k6 checks pass) but no linkage UPDATE fires.

**Impact:** Run #3 reported `linked_rows = 1` (not the formula-suggested ~150), which initially looked like a regression. Operator analysis confirmed the linkage path itself is correct — the fixture limited the count, not the runtime behavior. Sign-off accepted at `linked_rows > 0`. Future load tests: fixture-cycling (or fixture-multiplexing) is needed if "linkage rate" rather than "linkage works at all" is what's being tested.
**Source:** 02.1-08-SUMMARY.md (final run results note), 02.1-LOAD-TEST.md

### Worktree path confusion — first-pass writes hit main repo
First `Edit`/`Write` calls in plan 01 used absolute paths starting with the main repo root (`/Users/emoadm/projects/SMBsite/src/...`) instead of worktree-prefixed paths (`/Users/emoadm/projects/SMBsite/.claude/worktrees/agent-.../src/...`). Caught immediately by `git status` reporting "working tree clean" despite just-written files.

**Impact:** Reverted main-repo changes (Edit calls undid auth.ts/index.ts changes; `rm` on the orphan attribution.ts), then re-applied via worktree-prefixed paths. Net result: main repo untouched, worktree carried the canonical changes. No leakage to main.
**Source:** 02.1-01-SUMMARY.md (Issues Encountered §Initial accidental edits)
