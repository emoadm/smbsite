- **Two separate role concepts.** `users.role` keeps business-role values (D-10: 'owner'|'manager'|'employee'|'other') — do NOT touch. Payload `admin_users.role` is a SEPARATE table (`src/collections/Users.ts`) — keep both in sync via the ops runbook (delivered in Plan 04-07). PATTERNS.md "Critical Architecture Finding: Two Separate User Tables" is the authoritative explanation.
- **moderation_log is Drizzle-managed, NOT a Payload collection.** Payload UI on a moderation_log collection would expose Edit buttons and prevent DB-level REVOKE protection (Pitfall 1).
- **No raw `submitter_id`/`full_name`/`email` on any public surface.** D-C1 absolute lock — public schema exposure path is constrained at the query layer in Plan 04-05.
