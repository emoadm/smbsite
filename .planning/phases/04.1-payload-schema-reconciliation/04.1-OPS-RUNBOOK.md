# Phase 04.1 — Operations Runbook

> Procedures for the coalition operator to apply the missing Payload-side DDL
> (backfill SQL), re-deploy Phase 4 + Phase 02.3 cleanly, and recover if the
> re-deploy itself fails.
>
> This runbook is the canonical playbook for any future Payload schema change
> applied manually (until `.planning/todos/payload-tsx-esm-incompat.md` resolves
> and `payload migrate` can re-run in CI). Phase 3 — when it re-activates and
> adds voting fields to the `payload_ideas` collection — MUST reuse this
> procedure with a freshly-regenerated `canonical-schema.sql` and a new
> `.planning/ops/YYYY-MM-DD-payload-schema-phase-03-*.sql`.

2. The backfill SQL file exists locally:
   ```bash
   ls -l .planning/ops/2026-05-12-payload-schema-phase-04-backfill.sql
   ```

3. The local idempotence test from Plan 04.1-03 Task 2 passed (the SQL was
   applied twice against a local Postgres branch and the second run was a
   clean no-op — see `.planning/phases/04.1-payload-schema-reconciliation/04.1-03-SUMMARY.md`).

**CRITICAL:** Do NOT apply the backfill SQL if any of the above is missing.

## 3. Apply the backfill SQL

Open `.planning/ops/2026-05-12-payload-schema-phase-04-backfill.sql` in your
editor. Copy each STEP block separately into the Neon SQL Editor and run it.
Record outputs in `.planning/phases/04.1-payload-schema-reconciliation/04.1-APPLY-LOG.md`
as you go.

The SQL has **eight STEPs (0 through 7)**:
- STEP 0 — pre-apply verification SELECTs (read-only)
- STEP 1 — create new ENUMs (`enum_pages_status`, `enum_payload_ideas_status`, `enum_payload_ideas_topic`)
- STEP 2 — add `super_editor` value to `enum_admin_users_role`
- STEP 3 — create new tables (`pages`, `payload_ideas`, `payload_kv`)
- STEP 4 — add new FK columns on `payload_locked_documents_rels` (`pages_id`, `payload_ideas_id`)
- STEP 5 — create indexes
- STEP 6 — add FK constraints (DO-block)
- STEP 7 — post-apply verification SELECTs (read-only)

### Step 3.1 — STEP 0 (verification SELECTs, read-only)

Paste the entire STEP 0 block. Run each SELECT one at a time (0a–0f).

Expected outputs:

- **0a)** Table list contains the pre-Phase-4 tables (admin_users,
  admin_users_sessions, attribution_events, community_channels, consents,
  ideas (Drizzle), moderation_log, newsletters, payload_locked_documents,
  payload_locked_documents_rels, payload_migrations, payload_preferences,
  payload_preferences_rels, submissions, users, verification_tokens,
  accounts, sessions). NO `pages`, NO `payload_ideas`. May or may not
  contain `payload_kv` (depends on Payload version that booted into prod).
- **0b)** `payload_locked_documents_rels` columns: id, order, parent_id, path,
  admin_users_id, newsletters_id (six columns). NO `pages_id`,
  NO `payload_ideas_id` yet.
- **0c)** `payload_preferences_rels` columns: id, order, parent_id, path,
  admin_users_id (five columns). MUST stay these five columns post-apply too —
  the backfill does NOT touch this table.
- **0d)** Exactly two rows: `payload_locked_documents_rels` and
  `payload_preferences_rels`. Both single-underscore (`_rels`), NOT double
  (`__rels`).
- **0e)** `enum_admin_users_role` values pre-apply: `admin`, `editor` (two
  rows). If three rows already (including `super_editor`), the ADD VALUE in
  STEP 2 is a no-op — proceed.
- **0f)** Nine Drizzle-managed tables present: users, accounts, sessions,
  verification_tokens, consents, attribution_events, submissions,
  moderation_log, ideas.

**CRITICAL:** If 0a returns `pages` or `payload_ideas` already, STOP. The
idempotent CREATE blocks will skip them (safe), but the prod state may have
diverged from `canonical-schema.sql`. Record the deviation in APPLY-LOG.md
and surface back to the planner before continuing.

**CRITICAL:** If 0d returns a row with `__rels` (double underscore), STOP. The
backfill SQL was written for single-underscore naming (Payload v3 convention).
Surface back — the planner needs to revise the SQL.

**CRITICAL:** If 0f returns fewer than 9 rows, STOP. The prerequisite Drizzle
migration (0003) is missing or partial. Do NOT proceed with the Payload
backfill until the Drizzle ledger is reconciled.

### Step 3.2 — STEP 1 (CREATE TYPE for new ENUMs)

Paste STEP 1 (the DO-block that creates `enum_pages_status`,
`enum_payload_ideas_status`, `enum_payload_ideas_topic`). Run.

Expected: 0 errors. The DO block returns `DO`. If the types already exist
(re-run scenario), the IF NOT EXISTS guards skip them silently — no NOTICE is
emitted by the inner CREATE because the guard never enters the branch.

### Step 3.3 — STEP 2 (ALTER TYPE … ADD VALUE for super_editor)

Paste STEP 2. Run.

Expected: 0 errors. Returns `ALTER TYPE`. Idempotent via `ADD VALUE IF NOT
EXISTS` (Postgres 12+ syntax; Neon is 16).

### Step 3.4 — STEP 3 (CREATE TABLE for pages, payload_ideas, payload_kv)

Paste STEP 3 (blocks 3a, 3b, 3c). Run.

Expected: 0 errors. Each `CREATE SEQUENCE` returns `CREATE SEQUENCE` (or
NOTICE if it pre-existed). Each `CREATE TABLE` returns `CREATE TABLE` (or
NOTICE: relation already exists, skipping). Each `ALTER SEQUENCE … OWNED BY`
returns `ALTER SEQUENCE`.

### Step 3.5 — STEP 4 (ALTER TABLE … ADD COLUMN for pages_id, payload_ideas_id)

Paste STEP 4. Run.

Expected: 0 errors. Two `ALTER TABLE` results (or NOTICE if the columns
pre-existed).

**CRITICAL:** STEP 4 mutates ONLY `payload_locked_documents_rels`. It does NOT
touch `payload_preferences_rels`. Per 04.1-DELTA.md §5 / §11 D-1, Payload
only generates rels-FK columns for collections that the parent table actually
references — `payload_preferences` only references `admin_users`, so no
`pages_id` / `payload_ideas_id` columns are needed there. If you see the SQL
attempting to ALTER `payload_preferences_rels`, STOP — the SQL has been
corrupted in your editor.

### Step 3.6 — STEP 5 (CREATE INDEX)

Paste STEP 5 (blocks 5a–5d, totalling eight `CREATE INDEX` statements). Run.

Expected: 0 errors per CREATE INDEX. Each returns `CREATE INDEX` (or NOTICE
if the index already exists, skipping).

### Step 3.7 — STEP 6 (ADD CONSTRAINT for new FK columns)

Paste STEP 6 (the DO-block that adds `payload_locked_documents_rels_pages_fk`
and `payload_locked_documents_rels_ideas_fk`). Run.

Expected: 0 errors. The DO block returns `DO`. The two FK constraints are
added with `ON DELETE CASCADE`, matching the pattern of the pre-existing
`*_admin_users_fk` and `*_newsletters_fk` constraints.

**Note on constraint naming asymmetry:** The `payload_ideas_id` FK constraint
is named `…_ideas_fk` (NOT `…_payload_ideas_fk`) because Payload generates
constraint names from the collection SLUG (`ideas`), not its DBNAME
(`payload_ideas`). This is intentional — see 04.1-DELTA.md §11 D-2.

### Step 3.8 — STEP 7 (post-apply verification SELECTs)

Paste STEP 7. Run each SELECT (7a–7f).

Expected outputs:
- **7a)** Three rows, all returning `true` (or `t`): `pages`,
  `payload_ideas`, `payload_kv` all exist.
- **7b)** Exactly 2 rows on `payload_locked_documents_rels`:
  (`pages_id`, integer, YES), (`payload_ideas_id`, integer, YES).
- **7c)** Exactly **0 rows**. `payload_preferences_rels` MUST NOT have
  `pages_id` or `payload_ideas_id`. If non-empty, STOP — the backfill
  inadvertently mutated the wrong rels table.
- **7d)** Three rows in `enum_admin_users_role`: `admin`, `editor`,
  `super_editor`.
- **7e)** Five FK constraint rows on `payload_locked_documents_rels`:
  `payload_locked_documents_rels_admin_users_fk`,
  `payload_locked_documents_rels_ideas_fk` (NEW),
  `payload_locked_documents_rels_newsletters_fk`,
  `payload_locked_documents_rels_pages_fk` (NEW),
  `payload_locked_documents_rels_parent_fk`.
- **7f)** Nine Drizzle-managed table rows — same nine as 0f. The backfill
  MUST NOT touch any Drizzle table.

**CRITICAL:** If 7a, 7b, 7d, 7e, or 7f returns fewer rows than expected, or
if 7c returns ANY rows, STOP. Record in APPLY-LOG.md and surface back. Wave 5
(revert-the-revert + re-deploy) MUST NOT proceed with an incomplete or
incorrect schema.

## 4. Revert-the-revert + re-deploy (Wave 5 — covered in Plan 04.1-05)

Plan 04.1-05 handles this step end-to-end. The runbook section is kept here
for narrative completeness.

1. `git checkout main`

   `deaadc0` — this restores Phase 4 + Phase 02.3 code on main)
3. Push to main → deploy.yml fires.
4. **Approve the `migrate` job** in GitHub Actions:
   - Browser: <https://github.com/emoadm/SMBsite/actions>
   - Click the workflow run for the revert-of-revert commit.
   - "Review pending deployments" → environment `production` → Approve and deploy.
   - The `migrate` job is a no-op (Drizzle ledger already has 0003; no new
     migrations). It still requires approval per D-23 destructive gate.
5. `flyctl deploy` builds and rolls a new image (~5 min).
6. Post-deploy smoke gate runs automatically (GET `/`, `/register`, `/login`
   with >100B body floor + 3-attempt retry).

## 6. Rollback procedure (if Wave 5 re-deploy fails)

If the re-deploy crashes the admin shell AGAIN (unlikely — the backfill SQL
closes the known gap, but unknown unknowns happen):

1. Roll prod back to v52 image:
   ```bash
   flyctl deploy \
     --image registry.fly.io/smbsite-prod:deployment-01KR49EAD0V2BE2W8NSYT63V95 \
     --strategy rolling \
     --app smbsite-prod
   ```
   The v52 image was the last good pre-Phase-4 image (built from SHA `3e052f5`).
2. On main: re-apply the revert if it was removed by Wave 5:
   ```bash
   git revert <wave-5-revert-of-revert-sha> --no-edit
   git push origin main   # commit message should include [skip ci]
   ```
   (Alternative: leave main on the broken state, since the v52 image already
   serves prod; investigate before re-trying.)
3. The backfill SQL applied here is additive-only, so prod Neon stays in the
   post-apply state regardless of which image runs. The schema is forward-
   compatible: v52 (pre-Phase-4) does not consume the new tables, but their
   presence does no harm.
4. Surface to the planner. Likely root cause: a Phase 4 code path that
   references a column NOT in `canonical-schema.sql` (i.e., the dump missed
   something). The planner regenerates the dump on a fresh DB and audits.
