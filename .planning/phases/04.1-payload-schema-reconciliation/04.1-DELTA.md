- The Drizzle migrations on `main` (`0000_init.sql`, `0001_grey_umar.sql`,
  `0002_panoramic_ink.sql`) only touch Drizzle-managed tables. The
  Payload-managed tables in prod were created by Payload itself on first
  `push: true` runs against earlier `payload.config.ts` revisions.
- The Drizzle migration `0003_phase04_submissions.sql` (Phase 4 branch) was
  operator-applied directly to prod Neon via SQL Editor on 2026-05-10 (see
  `.planning/ops/2026-05-11-drizzle-ledger-backfill-0003.sql`). It created
  Drizzle's `submissions`, `moderation_log`, and Drizzle's own `ideas` table

**Delta type:** Additive only. Every item below must be CREATEd or
ADD-COLUMNed on prod. No DROP, no ALTER COLUMN of existing columns. The
backfill SQL at `.planning/ops/2026-05-12-payload-schema-phase-04-backfill.sql`
is the operator-apply artifact derived from this delta.

**Reading order:** Drizzle exclusions ‚Üí already-in-prod Payload tables ‚Üí new

## 3. ENUM value addition (`super_editor` on `enum_admin_users_role`)

Pre-Phase-4 `Users.ts` declared role options `[admin, editor]`. Phase 4
EDIT-02 added `super_editor`. Postgres `ALTER TYPE ‚Ä¶ ADD VALUE IF NOT EXISTS`
(PG 12+) is idempotent and is the correct operator-apply form.

| Operation | Target | New value | Source | canonical-schema.sql line |
| --- | --- | --- | --- | --- |
| ALTER TYPE ‚Ä¶ ADD VALUE IF NOT EXISTS | `public.enum_admin_users_role` | `'super_editor'` | Phase 4 EDIT-02 (Users.ts role select option) | 39‚Äì43 (canonical shows the post-Phase-4 enum with 3 values) |

> Note: `ALTER TYPE ‚Ä¶ ADD VALUE` cannot run inside a transaction block in
> earlier Postgres versions. Neon's Postgres 16 supports this since PG 12
> (the per-statement transaction rule was relaxed). The operator should
> run this statement OUTSIDE any explicit `BEGIN; ‚Ä¶ COMMIT;` block, which
> the backfill SQL handles by NOT wrapping the file in BEGIN/COMMIT.

## 4. New tables (CREATE TABLE IF NOT EXISTS)

Each table here has an associated `CREATE SEQUENCE` + `ALTER TABLE ‚Ä¶ SET
DEFAULT nextval(‚Ä¶)` + `ADD CONSTRAINT ‚Ä¶ PRIMARY KEY` block in
canonical-schema.sql. The backfill SQL idempotently re-creates each (sequence
via `CREATE SEQUENCE IF NOT EXISTS`, default via the column definition inside
`CREATE TABLE IF NOT EXISTS`, PK via inline `CONSTRAINT ‚Ä¶ PRIMARY KEY (id)`).

| Table | Origin (Payload collection) | DDL source (canonical-schema.sql lines) |
| --- | --- | --- |
| `public.pages` | Pages collection (Phase 4 EDIT-03) | 242‚Äì251 (table) + 258‚Äì264 (seq) + 533‚Äì536 (default) + 624‚Äì625 (pkey) |
| `public.payload_ideas` | Ideas collection (Phase 4 EDIT-02, Plan 04.1-01 `dbName: 'payload_ideas'` override) | 278‚Äì290 (table) + 297‚Äì303 (seq) + 542‚Äì543 (default) + 632‚Äì633 (pkey) |

> **No Lexical block sub-tables exist** (per Plan 04.1-02 SUMMARY Deviation #3).
> Payload v3.84 with the default `lexicalEditor()` stores richText fields
> (`pages.body`, `payload_ideas.description`, `newsletters.body`) as a single
> inline `jsonb NOT NULL` column rather than as relational block sub-tables.
> The backfill SQL therefore has NO `CREATE TABLE pages_body_*` or
> `CREATE TABLE payload_ideas_description_*` blocks.

> **No `pages_rels` / `payload_ideas_rels` tables exist** (verify via
> `grep -E '^CREATE TABLE public\.(pages|payload_ideas)_rels' canonical-schema.sql`
> ‚Üí no match). Neither collection declares a `relationship` field that would

## 5. New FK columns on existing rels tables (ALTER TABLE ‚Ä¶ ADD COLUMN IF NOT EXISTS)

The originally-failing query in the prod log was:

```
select "payload_locked_documents"."id", ‚Ä¶ "payload_locked_documents__rels"."pages_id" ‚Ä¶
```

Pre-Phase-4 `payload_locked_documents_rels` had FK columns for only the two
pre-Phase-4 user/global collections: `admin_users_id`, `newsletters_id`.
Phase 4 added two more collections (`Pages`, `Ideas`), which causes Payload to
add two new nullable `integer` FK columns to the locked-documents rels join.

`payload_preferences_rels` is asymmetric ‚Äî it only joins back to admin_users
(the user whose preferences are stored). Per the canonical-schema.sql dump
(line 482‚Äì488), `payload_preferences_rels` has columns `id, "order",
parent_id, path, admin_users_id` only. **No `pages_id` or `payload_ideas_id`
column is added to `payload_preferences_rels`.** (This is the asymmetry
correction relative to the plan's initial assumption ‚Äî see ¬ß10 Open
Verification 2 + ¬ß11 Deviation.)

| Table | Column | Type | Nullable? | canonical-schema.sql ref |
| --- | --- | --- | --- | --- |
| `payload_locked_documents_rels` | `pages_id` | `integer` | YES | line 387 |
| `payload_locked_documents_rels` | `payload_ideas_id` | `integer` | YES | line 388 |

That is the entire ALTER-TABLE delta. Two columns on one table.

> **Why naming `payload_locked_documents_rels` (single `_`) not `__rels`
> (double `_`)?** The Fly error log printed `payload_locked_documents__rels.pages_id`
> with double underscore (CONTEXT.md "References" section). The canonical
> dump shows single underscore (`payload_locked_documents_rels`). The
> double-underscore form is the original Payload v1/v2 convention; Payload
> v3 normalized this to single underscore. The dump is the source of truth
> (Payload v3.84 generated it). The error log line was likely from an
> intermediate query-builder string; the actual prod table is single
> underscore (Payload v3 has been running in prod since pre-Phase-4 era ‚Äî
> verify in Step 0d of the backfill SQL).

Each new table gets its own `id_seq` per Payload's `integer` PK convention.
The backfill SQL uses `CREATE SEQUENCE IF NOT EXISTS` (PG 9.5+) which is
idempotent.

| Sequence | Owner table.column | canonical-schema.sql line |
| --- | --- | --- |
| `public.pages_id_seq` | `pages.id` | 258‚Äì264 |
| `public.payload_ideas_id_seq` | `payload_ideas.id` | 297‚Äì303 |
| `public.payload_kv_id_seq` (if `payload_kv` newly-created) | `payload_kv.id` | 328‚Äì334 |

> **Setting the SEQUENCE OWNED BY** (canonical lines 270, 309, 340 for the
> three new sequences): the backfill SQL emits `ALTER SEQUENCE ‚Ä¶ OWNED BY
> table.col`. This is idempotent at the Postgres level (re-issuing it is a
> no-op when the ownership already matches), so no DO-block guard is needed.

## 9. NOT-NULL defaults on PK columns (`ALTER COLUMN id SET DEFAULT nextval(‚Ä¶)`)

Each new table's `id` column needs a `DEFAULT nextval('public.<table>_id_seq'::regclass)`
to mirror canonical-schema.sql lines 515 / 536 / 543 / 550. The backfill SQL
sets this via the `CREATE TABLE IF NOT EXISTS` block's inline column default
(`id integer NOT NULL DEFAULT nextval(‚Ä¶)`), which is idempotent because
`CREATE TABLE IF NOT EXISTS` is a no-op if the table already exists. This is
**not** a separately tracked delta item ‚Äî it's part of the ¬ß4 CREATE TABLE
operation.

## 10. Open questions / required Step-0 verifications

For each item below, the backfill SQL Step 0 must include a SELECT that
confirms the assumption. If any assumption is FALSE, the operator pauses and
surfaces back.

1. **`__rels` vs `_rels` naming.** CONTEXT origin section references
   `payload_locked_documents__rels.pages_id` (double underscore). The canonical
   dump shows single underscore. **Step 0d query:** list all tables matching
   either pattern. Expected: only single-underscore versions exist in prod.

2. **`payload_preferences_rels` column set.** The original plan suggested adding
   `pages_id` and `payload_ideas_id` to BOTH rels tables. The dump confirms
   they are added ONLY to `payload_locked_documents_rels`, NOT to
   `payload_preferences_rels`. **Step 0c query:** list all columns of
   `payload_preferences_rels`. Expected (pre-apply AND post-apply, unchanged):
   `id, order, parent_id, path, admin_users_id`. If the dump's
   `payload_preferences_rels` differs from prod, the assumption is wrong.

3. **`admin_users.role` ENUM values in prod.** Pre-Phase-4 had `[admin,
   editor]`. The backfill must `ALTER TYPE ‚Ä¶ ADD VALUE IF NOT EXISTS
   'super_editor'`. **Step 0e query:** `SELECT enumlabel FROM pg_enum WHERE
   enumtypid = 'public.enum_admin_users_role'::regtype ORDER BY enumsortorder;`
   Expected pre-apply: `admin, editor`. Expected post-apply: `admin, editor,
   super_editor`.

4. **Does `payload_kv` already exist in prod?** Payload v3.84 creates it on
   first boot, but prod's Payload version at the time of v52 deploy may have
   been earlier (pre-`payload_kv` introduction). **Step 0a query:** the full
   `information_schema.tables` listing will show whether `payload_kv` is in
   prod. The backfill's `CREATE TABLE IF NOT EXISTS public.payload_kv`
   handles both cases.

5. **Are there any unexpected Payload-managed `_v` (versions) sub-tables in
   prod?** No collection declares `versions: true`, so the dump shows none.
   But if prod has lingering `_v` tables from a long-ago Payload version that
   had drafts enabled and was later reverted, the backfill's `CREATE TABLE IF
   NOT EXISTS` won't touch them ‚Äî they're inert. **Step 0a query** (the full
   table list) makes any such tables visible to the operator for awareness;
   no action required.

6. **Is the prod `payload_locked_documents_rels.path` column NOT NULL?** The
   canonical dump shows `path character varying NOT NULL`. The backfill's
   ALTER TABLE ADD COLUMN adds two NEW columns (`pages_id`, `payload_ideas_id`)
   both **nullable** integers. **Step 0b query:** `is_nullable=YES` for both
   new columns is the expected post-apply state, matching canonical.

## 11. Deviations from plan-level expectations

### D-1. `payload_preferences_rels` does NOT get `pages_id` / `payload_ideas_id` columns

- **Plan-level expectation** (04.1-03-PLAN.md `<context>` ‚Üí "compute the delta"
  step 4 + acceptance criterion): both `payload_locked_documents_rels` AND
  `payload_preferences_rels` get new `pages_id` + `payload_ideas_id` columns
  (4 ALTER TABLE statements total).
- **Canonical truth:** Only `payload_locked_documents_rels` gets the new FK
  columns. `payload_preferences_rels` retains its pre-Phase-4 column set
  (`id, "order", parent_id, path, admin_users_id`).
- **Why:** `payload_preferences` is a per-admin-user UI preferences store. Its
  rels join only ever points back to `admin_users` (preferences are owned by
  a single admin user; they don't reference other collections). Payload v3.84
  generates FK columns on a rels table only for collections that the parent
  table can reference; `payload_preferences` references only `admin_users`,
  so its rels table has only `admin_users_id`.
- **Risk:** None ‚Äî the canonical dump is the source of truth (Wave 2
  Deviation #4 already noted the symmetric finding for globals). The plan's
  initial estimate was conservative; the dump corrected it. The backfill SQL
  reflects the canonical truth (2 ALTER TABLE statements, not 4).

### D-2. Constraint name uses `_ideas_fk`, not `_payload_ideas_fk`

- **Plan-level expectation** (PLAN.md STEP 3 example DO-block): constraint
  name `payload_locked_documents_rels_payload_ideas_fk`.
- **Canonical truth:** name is `payload_locked_documents_rels_ideas_fk` (Payload
  generates FK names from the collection *slug*, not *dbName*).
- **Why:** Plan 04.1-01 kept slug `'ideas'` (admin URL preserved) and only
  added `dbName: 'payload_ideas'`. The FK-name generator reads slug.
- **Risk:** None ‚Äî the backfill SQL uses the exact canonical name so future
  `pg_dump` comparisons match.

### D-3. `ALTER TYPE enum_admin_users_role ADD VALUE 'super_editor'` is part of the delta

- **Plan-level expectation** (PLAN.md `<critical_constraints>`): "The backfill
  is additive-only" ‚Äî implicit only mentioned tables/columns/indexes, no
  ENUMs.
- **Canonical truth:** Phase 4 added `super_editor` to the role enum. The
  Users collection in `gsd/phase-04-‚Ä¶` HEAD has three role options
  (`admin`, `editor`, `super_editor`); pre-Phase-4 had two. Prod's
  `enum_admin_users_role` currently has only `admin, editor`.
- **Why:** Without this ALTER TYPE, future `INSERT ‚Ä¶ role='super_editor'`
  from the moderation-queue admin UI will fail with "invalid input value for
  enum".
- **Risk:** None ‚Äî `ALTER TYPE ‚Ä¶ ADD VALUE IF NOT EXISTS` is idempotent on
  Postgres 12+ (Neon is 16). Documented as Open Verification 3.
