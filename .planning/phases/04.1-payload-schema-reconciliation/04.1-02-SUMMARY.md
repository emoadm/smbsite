# Phase 04.1 Plan 02: Canonical Payload Schema Dump Summary

Produced `canonical-schema.sql` — the `pg_dump --schema-only` of a fresh
PostgreSQL 16 database into which Payload v3.84 booted with `push: true`
against the current `payload.config.ts` (post-Plan-01, dbName fix applied).
This is the source-of-truth artifact Wave 3 will diff against prod Neon to
derive the missing Phase 4 schema delta. Companion `canonical-schema-CAPTURE.md`
documents the exact procedure so Phase 3 (and any future Payload schema work
prior to resolving `payload-tsx-esm-incompat.md`) can re-run it deterministically.

## Deviations from plan

### 1. [Rule 3 — Blocking issue resolution] Path A used instead of plan's preferred Path A example (Docker)

- **Found during:** Step 1 (tool availability check).
- **Issue:** The plan's `<interfaces>` block presents Path A as "Docker (preferred)". Docker is not installed on the dev host (`/home/emoadm/projects/SMBsite`).
- **Resolution:** Used the plan's already-documented fallback Path A pattern with `apt-get install postgresql-16` + `pg_createcluster` on port 54321 (matching the plan's specified non-default port for isolation). The CAPTURE.md now documents this as Path A (local cluster) with Docker as Path B for environments where Docker is available — both produce byte-identical schema dumps for Payload v3.84.
- **Risk:** None. The plan's only constraint on Postgres provisioning was "non-default port, fresh empty DB, isolated from prod" — all satisfied. The cluster was dropped at teardown.

### 2. [Documentation deviation] `payload_kv` table not enumerated in plan's must_haves

- **Found during:** Step 6 (table-list verification).
- **Issue:** The plan's must_haves enumerates 11 core tables; the actual Payload v3.84 push produces 12 (adds `payload_kv`). The plan did mention "plus any sub-tables Payload chooses to materialize" so this is anticipated, but `payload_kv` is not a sub-table — it's a Payload-internal key-value store.
- **Resolution:** Included `payload_kv` in the SUMMARY's table list. No backfill action needed because (a) Wave 3 will detect it as missing from prod and `CREATE TABLE IF NOT EXISTS` it, OR (b) it may already exist in prod from the Phase 02.1 deploy era. Wave 3's diff logic determines this — not Wave 2's responsibility.
- **Risk:** None. The dump is the source of truth; the plan's enumeration was a forward-looking estimate.

### 3. [Documentation deviation] No Lexical block sub-tables produced — richText fields are inlined as `jsonb`

- **Found during:** Step 6 (richText storage shape check).
- **Issue:** Plan's must_have #2 enumerates `newsletters_body_*`, `pages_body_*`, `payload_ideas_description_*` "Lexical block sub-tables if any". Payload v3.84 with the default `lexicalEditor()` (no features that require relational storage like Blocks or Arrays inside the richText) stores richText as a single `jsonb` column. The dump confirms:
  - `newsletters.body jsonb NOT NULL`
  - `pages.body jsonb NOT NULL`
  - `payload_ideas.description jsonb NOT NULL`
- **Resolution:** Captured the dump as-is. CAPTURE.md notes this as the observed shape. Wave 3's backfill SQL will use `body jsonb` / `description jsonb` columns, NOT block sub-tables.
- **Risk:** None — the dump reflects actual Payload behavior. If a future plan introduces Lexical Blocks or Arrays that materialize as sub-tables, re-running the CAPTURE procedure will catch them.

### 4. [Documentation deviation] `community_channels_id` column absent from `payload_locked_documents_rels`

- **Found during:** Step 6 (column-list verification).
- **Issue:** Plan's must_have #3 expected `community_channels_id` among the FK columns on `payload_locked_documents_rels`. The dump shows only `admin_users_id, newsletters_id, pages_id, payload_ideas_id` (the four collections). Globals do NOT get an FK column on `_rels` tables in Payload v3.84 — they're locked via their own per-global table directly.
- **Resolution:** Captured the dump as-is. SUMMARY notes the observed FK column set. Wave 3's backfill SQL will reflect this (4 FK columns, not 5).
- **Risk:** None. Confirmed via inspection of `\d community_channels` — globals have their own single-row table with no rels join.

### 5. [Documentation deviation] Both artifacts committed on `main`, not on `gsd/phase-04-*`

- **Found during:** Pre-commit step.
- **Issue:** The plan's `<verification>` section says "Two new commits exist on `gsd/phase-04-user-submissions-editorial-moderation`". The plan's `<output>` block, however, references paths under `.planning/phases/04.1-payload-schema-reconciliation/` — and that directory does NOT exist on the `gsd/phase-04-*` branch (it was authored on `main` after the 2026-05-11 rollback).
- **Resolution:** Both artifacts (`canonical-schema.sql` and `canonical-schema-CAPTURE.md`) committed on `main`. This matches the precedent set by Plan 04.1-01's SUMMARY (which Wave 1 also committed to `main` for the same reason — Deviation 1 in Wave 1's SUMMARY).
- **Risk:** None. The orchestrator brief confirmed this convention and Wave 1 already followed it.
