## Origin

PR #2 (Phase 4 + Phase 02.3, merge commit `deaadc0`) deployed to chastnik.eu
2026-05-11 23:01 UTC. Within 25 minutes, every authenticated path under `/admin`
threw a Next.js server-side exception. Root cause:

```
column payload_locked_documents__rels.pages_id does not exist
```

Phase 4 added two new Payload collections (`Pages`, `Ideas`) to
`payload.config.ts:42`. Per Payload v3 convention, each new collection requires
a corresponding `<collection>_id` FK column on every `*__rels` join table
(`payload_locked_documents__rels`, `payload_preferences__rels`, etc.), plus the
collection's own table + its `_rels` join + any block/array sub-tables. None
of that ran because `payload migrate` is **disabled in `deploy.yml`** (see
`.planning/todos/payload-tsx-esm-incompat.md` and project memory
`project_payload_schema_constraint.md` â€” tsx@4.21 + Node 22 ESM incompat).

Drizzle migration `0003_phase04_submissions.sql` ran fine (operator-applied via
Neon SQL Editor 2026-05-10), but it covered only the *Drizzle-managed* tables
(`submissions`, `moderation_log`, `ideas`). The *Payload-managed* schema delta
for the new collections was never written.

Rollback to v52 (image `:deployment-01KR49EAD0V2BE2W8NSYT63V95`, pre-merge SHA
`3e052f5`) restored `/admin` at 2026-05-12 00:42 UTC. Merge commit `deaadc0`
was reverted on main with `[skip ci]` to keep git and prod aligned without
re-firing the broken deploy.

     Drizzle's `ideas`), plus any sub-tables for blocks/arrays in those
     collections.
   - New columns on existing `*__rels` join tables: `pages_id`, `ideas_id` on

   - (a) Rename Payload's collection to use `dbName: 'payload_ideas'` (cleaner;
