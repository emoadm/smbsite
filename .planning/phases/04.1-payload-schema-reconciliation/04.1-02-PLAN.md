<objective>
Generate the canonical Payload-managed schema for the current `payload.config.ts` (Users [admin_users slug], Newsletters, Pages, Ideas with dbName=payload_ideas, CommunityChannels global) by booting Payload locally against a fresh Postgres database with `push: true`, then capturing the resulting DDL via `pg_dump --schema-only`. Commit the dump to `.planning/phases/04.1-payload-schema-reconciliation/canonical-schema.sql` as the source-of-truth for Wave 3's diff against prod Neon.

Purpose: Phase 4 added two new Payload collections (Pages, Ideas) without writing their DDL to prod (because `payload migrate` is disabled in CI per `.planning/todos/payload-tsx-esm-incompat.md`). To know what prod is missing, we need a deterministic dump of what Payload WANTS the schema to look like. This dump becomes the reference for every future Payload schema change until the tsx-ESM incompat is resolved.

Output: Two files committed to the Phase 04.1 directory — the schema dump itself and a reproducible-procedure note.
</objective>

**Path B (fallback — Neon dev branch):**

Payload boot for schema-push:

```bash

# Or set PAYLOAD_FORCE_PUSH=true and confirm payload.config.ts respects it (currently it does not — see below).

pnpm exec next dev 2>&1 | tee /tmp/payload-push.log
# Wait until the log shows "Server initialized" (~20-40s for fresh-DB Payload boot with push).
# Ctrl-C after Payload reports "Successfully connected to the database" AND no migration errors.
```

pg_dump capture:

```bash

  --schema-only \
  --no-owner \
  --no-acl \
  --schema=public \
  postgresql://postgres:local@127.0.0.1:54321/payload_canonical \
  > .planning/phases/04.1-payload-schema-reconciliation/canonical-schema.sql
```

Cleanup:

```bash

# OR for Path B: drop the Neon dev branch via Neon dashboard
```
