# Phase 04.1 Plan 03: Schema Delta + Idempotent Backfill SQL Summary

Produced the two operator-facing artifacts for Wave 4 — `04.1-DELTA.md`
(human-readable diff enumeration) and `2026-05-12-payload-schema-phase-04-backfill.sql`
(idempotent paste-ready DDL for Neon SQL Editor). Both files mirror the
shape of the prior `.planning/ops/2026-05-11-drizzle-ledger-backfill-0003.sql`
pattern (STEP 0 read-only verification SELECTs → STEP 1-N idempotent DDL →
STEP 7 read-only post-apply verification SELECTs). The backfill SQL was
validated by a 2x apply against a freshly-seeded PG16 local database: both
runs returned exit code 0; post-apply state was byte-identical.

## Approach used

**Deductive (per orchestrator brief option 1).** Live prod read access was
not available (the project's `.neon` file at repo root contains only an
organization pointer JSON `{"orgId":"org-shiny-mode-87914159"}` — no DSN, no
endpoint). The delta was therefore derived from:

1. **Canonical truth = `canonical-schema.sql`** (Wave 2 dump, 977 lines,
   25 KB, committed at `6d69f94`). This is the *target* schema — what prod
   must contain after Wave 4 applies the backfill.
2. **Baseline truth = pre-Phase-4 main HEAD `3e052f5`** (the SHA prod
   rolled back to as Fly image v52). Inspected the live files via
   `git show 3e052f5:<path>`:
   - `src/payload.config.ts` — confirmed pre-Phase-4 had only
     `collections: [Users, Newsletters]` + `globals: [CommunityChannels]`.
     So pre-Phase-4 Payload tables in prod = `admin_users`,
     `admin_users_sessions`, `newsletters`, `community_channels`, plus the
     5 Payload built-ins (`payload_locked_documents`,
     `payload_locked_documents_rels`, `payload_preferences`,
     `payload_preferences_rels`, `payload_migrations`).
   - `src/collections/Users.ts` — pre-Phase-4 role enum was
     `[admin, editor]` (Phase 4 added `super_editor`).
3. **Drizzle baseline = `src/db/migrations/0000-0002` on main + the
   operator-applied 0003** (per the 2026-05-11 ledger-backfill SQL).
   Drizzle-owned tables in prod = `users`, `accounts`, `sessions`,
   `verification_tokens`, `consents`, `attribution_events`, `submissions`,
   `moderation_log`, `ideas`.

The backfill SQL's Step 0 verification SELECTs embed every assumption above
as a queryable check, so the operator at Wave 4 apply time independently
re-confirms the deductive baseline before issuing any DDL. If any
assumption is false, the operator stops and surfaces back; the IF NOT EXISTS
guards on the subsequent DDL prevent corruption either way.

## Structure of the backfill SQL (`2026-05-12-payload-schema-phase-04-backfill.sql`)

| Step | Purpose | Idempotence mechanism | Statement count |
| --- | --- | --- | --- |
| Step 0 | Read-only pre-state verification SELECTs (Step 0a–0f) — operator confirms expectations before any DDL | n/a (read-only) | 6 SELECTs |
| Step 1 | Create new ENUMs | DO-block guard against pg_catalog.pg_type | 3 CREATE TYPE inside 1 DO-block |
| Step 2 | ENUM value addition | `ADD VALUE IF NOT EXISTS` (PG 12+) | 1 ALTER TYPE |
| Step 3 | Sequences + tables + sequence ownership | `CREATE SEQUENCE IF NOT EXISTS`, `CREATE TABLE IF NOT EXISTS`, idempotent `ALTER SEQUENCE … OWNED BY` | 3 sequences + 3 tables + 3 ownership |
| Step 4 | New FK columns | `ALTER TABLE … ADD COLUMN IF NOT EXISTS` | 2 ALTER TABLE |
| Step 5 | Indexes | `CREATE [UNIQUE] INDEX IF NOT EXISTS` | 8 CREATE INDEX |
| Step 6 | FK constraints | DO-block guard against information_schema.table_constraints | 2 ALTER TABLE inside 1 DO-block |
| Step 7 | Read-only post-apply verification SELECTs (Step 7a–7f) | n/a (read-only) | 6 SELECTs |

File header documents context (rollback incident origin), source-of-truth
references, idempotence rationale, transaction-shape rationale (not wrapped
in BEGIN/COMMIT because ALTER TYPE … ADD VALUE can be restricted inside
transactions), and an explicit Drizzle DO-NOT-TOUCH list.

## Idempotence proof — 2x apply against PG16

Environment: Ubuntu PG16 cluster (`pg_lsclusters` → `16 main 5432 online
postgres`), database `payload_backfill_check` (throwaway, dropped at
teardown), seeded with a synthetic pre-Phase-4 schema fragment that included
all 9 Drizzle-owned tables + the 9 pre-Phase-4 Payload tables (admin_users,
admin_users_sessions, newsletters, community_channels,
payload_locked_documents, payload_locked_documents_rels, payload_preferences,
payload_preferences_rels, payload_migrations) + the pre-Phase-4 role enum
`(admin, editor)`.

| Run | Command | Exit code | Outcome |
| --- | --- | --- | --- |
| Pre-seed | `psql -d payload_backfill_check -v ON_ERROR_STOP=on -f pre-phase-4-seed.sql` | 0 | 18 tables in `public.*` (9 Drizzle + 9 Payload), 3 enums, no Phase 4 objects |
| **APPLY 1** | `psql -d payload_backfill_check -v ON_ERROR_STOP=on -f backfill.sql` | **0** | 21 tables (18 + pages + payload_ideas + payload_kv), 6 enums, 8 new indexes, 2 new FK constraints, +1 new enum value. Step 7 verification SELECTs all return expected shapes. |
| **APPLY 2** (idempotence) | same command, against the post-APPLY-1 state | **0** | NO state changes. NOTICE messages emitted for the IF-NOT-EXISTS-guarded objects ("relation already exists, skipping"). DO-blocks silently skip pre-existing ENUMs + FK constraints. Step 7 verification SELECTs return BYTE-IDENTICAL rows to APPLY 1. |

Both runs exit cleanly under `ON_ERROR_STOP=on` (so a non-zero exit would
have aborted on the first failing statement — no statement failed).

## Deviations from plan

### D-1. [Documentation deviation] PLAN expected 4 ADD COLUMN statements; canonical truth is 2

- **Found during:** Task 1 — reading canonical-schema.sql lines 482–488
  (`payload_preferences_rels` table definition).
- **Issue:** PLAN.md `<context>` "compute the delta" step 4 stated:
  "Same [add `pages_id`, `payload_ideas_id`] for `payload_preferences_rels`."
  PLAN.md `<acceptance_criteria>` expected
  `grep -c "ADD COLUMN IF NOT EXISTS pages_id" file` >= 2 (one per rels
  table) and same for `payload_ideas_id` >= 2. Canonical-schema.sql shows
  `payload_preferences_rels` has only 5 columns (`id, "order", parent_id,
  path, admin_users_id`) — NO pages_id, NO payload_ideas_id, NO
  newsletters_id, NO community_channels_id.
- **Resolution:** DELTA.md §5 + §11 D-1 explicitly enumerate the asymmetry
  with the canonical-schema.sql line citation. Backfill SQL Step 4 contains
  exactly 2 ALTER TABLE ADD COLUMN statements (both targeting
  `payload_locked_documents_rels`); Step 0c + Step 7c embed assertions that
  `payload_preferences_rels` is unchanged pre- and post-apply.
- **Why the canonical truth is right:** Payload v3.84 only materializes FK
  columns on a `*_rels` table for collections that the parent table can
  reference. `payload_locked_documents` can lock documents in any
  collection (so its rels gets one FK column per collection). But
  `payload_preferences` only stores per-admin-user UI preferences — its
  rels join only ever needs `admin_users_id`.
- **Risk:** None. Wave 2's dump is the empirical record of what Payload
  actually generates; Wave 3 simply reflects it.

### D-2. [Documentation deviation] FK constraint name is `_ideas_fk`, not `_payload_ideas_fk`

- **Found during:** Task 2 Step 6 — when authoring the DO-block FK guards.
- **Issue:** PLAN.md STEP 3 example DO-block used the name
  `payload_locked_documents_rels_payload_ideas_fk`. Canonical-schema.sql
  line 925–929 shows the actual constraint name is
  `payload_locked_documents_rels_ideas_fk` (the table-suffix segment is the
  collection's **slug** — `ideas` — not its **dbName** — `payload_ideas`).
- **Resolution:** Backfill SQL Step 6 uses the canonical name verbatim.
  DELTA §7 + §11 D-2 explain the slug-vs-dbName-naming asymmetry.
- **Why this matters:** If we used the PLAN's example name
  (`_payload_ideas_fk`), a future `pg_dump` comparison against prod would
  show a constraint-name drift, which would either (a) trigger a spurious
  rollback in CI, or (b) leave prod with TWO different-named FKs both
  pointing at the same columns. Both are bad.

### D-3. [Documentation deviation] `ALTER TYPE … ADD VALUE 'super_editor'` is part of the delta

- **Found during:** Task 1 — comparing `git show 3e052f5:src/collections/Users.ts`
  (pre-Phase-4 role select: `[admin, editor]`) against
  `git show gsd/phase-04-…:src/collections/Users.ts` (Phase 4 branch:
  `[admin, editor, super_editor]`).
- **Issue:** PLAN.md only discussed table/column/index/FK deltas, did NOT
  enumerate ENUM-value additions.
- **Resolution:** DELTA §3 + §11 D-3 add a dedicated ENUM-value-addition
  section. Backfill SQL Step 2 issues
  `ALTER TYPE public.enum_admin_users_role ADD VALUE IF NOT EXISTS 'super_editor'`.
- **Why it matters:** Without this ALTER, the moderation-queue admin UI's
  "promote to super_editor" action will fail with `invalid input value for
  enum` on prod. This is part of the originally-failing deploy.

### D-4. [Documentation deviation] `payload_kv` included as a defensive CREATE

- **Found during:** Task 2 — Wave 2 SUMMARY Deviation #2 already flagged
  `payload_kv` as not in the plan's expected list. Plan 04.1-02 deferred
  the decision ("Wave 3's diff logic determines this — not Wave 2's
  responsibility").
- **Resolution:** Backfill SQL Step 3c emits `CREATE TABLE IF NOT EXISTS
  public.payload_kv` as a defensive idempotent CREATE. If prod already has
  it (likely — Payload v3.84 created it earlier), the CREATE is a no-op. If
  prod doesn't have it, it's created.
- **Risk:** None. DELTA §10 Open Verification 4 documents the uncertainty;
  Step 0a (full table list) makes the actual prod state visible to the
  operator at apply time.

### D-5. [Process deviation] Both artifacts committed to `main`, not `gsd/phase-04-*`

- **Issue:** PLAN.md's `<verification>` section says "Both `04.1-DELTA.md`
  and `2026-05-12-payload-schema-phase-04-backfill.sql` committed on
  `gsd/phase-04-user-submissions-editorial-moderation`". But the phase
  directory only exists on `main` (it was authored after the revert,
  commits `d6e843f`, `9c566d0`, `a42b779`), and `.planning/ops/` exists on
  `main` only.
- **Resolution:** Both files committed on `main` — same convention as Plan
  04.1-01 Deviation D-1 and Plan 04.1-02 Deviation #5. The orchestrator
  brief explicitly noted this convention ("Both files are .planning/ +
  .planning/ops/ artifacts — commit on `main`").
- **Risk:** None. The artifacts are planning/ops documents, not source
  code; they need to be on `main` so Wave 4's operator (who works from
  `main`) can paste them.
