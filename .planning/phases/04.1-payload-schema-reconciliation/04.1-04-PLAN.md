Rollback path (if the apply itself fails — which is unlikely given the idempotence design):
- The backfill SQL is additive-only. There is NO destructive change to roll back.
- If a CREATE TABLE fails halfway (e.g., disk-full, permission error), the partial state is recoverable via re-running the same SQL (IF NOT EXISTS handles partial CREATE; DO-block handles partial constraints).
- If a worse-than-expected state surfaces (e.g., schema corruption), the prod is ALREADY on v52 image — the application is not consuming the partial schema. Time pressure is low. Surface back, reassess.
